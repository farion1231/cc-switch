## 1. 产品定位

### 一句话描述

**OmniAgent Workbench** 是一个运行在 PC 本地的智能网关软件，对外提供标准 OpenAI/Anthropic 兼容 API，内部通过多模态路由、跨模态编排、级联升级、多候选融合、工具验证闭环，让 Claude Code / OpenCode / Codex 等 AI 编程客户端**透明无感地获得远超单一顶级模型的能力**。

### 核心价值主张

```
单一 GPT-5 / Claude Opus 4.7:
  你得到的是"一个模型的最高水平"

OmniAgent Workbench:
  你得到的是"一个工程系统的最高水平"
  = 多模态专长模型群 + 跨模态编排 + 工具验证 + 动态路由 + 成本优化
```

### 产品形态

一个 Windows/macOS/Linux 桌面应用（Tauri 2.x + Rust + React）：

- **前台 GUI**：模型管理、工作流可视化、成本仪表盘、评测面板、人工审批
- **后台 Gateway**：本地 OpenAI/Anthropic 兼容 API 服务 (localhost:15721)，透明代理所有请求
- **编排引擎**：动态工作流图，根据任务实时组合路由/级联/MoA/辩论/跨模态流水线

### 与 CC-Switch 的关系

CC-Switch 是本项目的重要参考和底座：
- **复用**：CC-Switch 的 provider 管理、proxy 模块、SQLite 存储、Tauri 架构
- **增强**：在 CC-Switch 之上新增智能编排引擎、跨模态调度、质量验证、评测系统
- **定位差异**：CC-Switch 是"配置切换器 + 反向代理"；OmniAgent Workbench 是"智能编排网关 + 跨模态操作系统"

### 与 MiroFish 的关系

MiroFish 是一个独立的 Python/Flask 社会模拟预测引擎。OmniAgent Workbench **移植**其已验证的架构模式到 Rust：
- **移植**：6态生命周期状态机、ReACT 循环、知识图谱 schema 生成、JSONL 审计、增量输出
- **不移植**：OASIS 仿真、Zep Cloud、Twitter/Reddit 平台脚本、Vue 前端
- **不依赖**：无 Python 运行时依赖，无 MiroFish 代码引用

## 3. 编排引擎 (Orchestration Engine) — 核心模块

### 3.1 请求处理全流程

```
外部客户端请求 (Claude Code / OpenCode / Codex)
  │
  ▼
┌──────────────────────────────────────────────────────────────┐
│ Step 1: Request Interceptor                                  │
│   识别来源客户端、请求类型、附带的工具调用、上下文大小         │
│   解析出: {client, messages, tools, model_preference, ...}    │
└──────────────────────────┬───────────────────────────────────┘
                           ▼
┌──────────────────────────────────────────────────────────────┐
│ Step 2: Modality Router                                      │
│   分析请求需要哪些模态参与:                                    │
│   - 有图片输入? → need_vision                                 │
│   - 有音频输入? → need_speech                                 │
│   - 需要代码输出? → need_code                                  │
│   - 需要工具调用? → need_tools                                 │
│   输出: ModalityPlan { required: [], optional: [], order: [] }│
└──────────────────────────┬───────────────────────────────────┘
                           ▼
┌──────────────────────────────────────────────────────────────┐
│ Step 3: Task Difficulty Classifier                            │
│   估算任务复杂度和风险:                                        │
│   - complexity: 0-1 (上下文长度、推理链深度、工具数量)         │
│   - risk: low/medium/high/critical                            │
│   - verifiability: 0-1 (是否有外部工具可验证)                  │
│   - ambiguity: 0-1 (需求是否模糊)                              │
└──────────────────────────┬───────────────────────────────────┘
                           ▼
┌──────────────────────────────────────────────────────────────┐
│ Step 4: Strategy Selector                                     │
│   根据分类结果选择策略:                                        │
│                                                                │
│   ┌──────────────┬─────────────────────────────────────────┐ │
│   │  条件         │  策略                                    │ │
│   ├──────────────┼─────────────────────────────────────────┤ │
│   │ 简单+低风险   │  ROUTE → 直接路由到最便宜的合适模型      │ │
│   │ 中等+可验证   │  CASCADE → 便宜先试，不确定再升级        │ │
│   │ 复杂+多模态   │  CROSS-MODAL → 模态分离，独立专家执行    │ │
│   │ 高风险+事实   │  DEBATE → 多模型独立回答 + 强Judge仲裁   │ │
│   │ 极高+抽象     │  MoA → 多Proposer + 强Aggregator        │ │
│   │ 代码+测试     │  CASCADE (带 test_runner 工具验证)       │ │
│   │ 模态创作      │  CROSS-MODAL (顺序模态转换)              │ │
│   └──────────────┴─────────────────────────────────────────┘ │
└──────────────────────────┬───────────────────────────────────┘
                           ▼
┌──────────────────────────────────────────────────────────────┐
│ Step 5: Workflow Executor (动态 DAG)                          │
│   构建执行图（节点 = 模型调用/工具调用/MCP调用，边 = 依赖）   │
│   支持动态修改（某个节点失败后插入修复节点）                    │
│   并行执行无依赖的节点                                         │
└──────────────────────────┬───────────────────────────────────┘
                           ▼
┌──────────────────────────────────────────────────────────────┐
│ Step 6: Cross-Modal Aggregator                               │
│   合并多模态路径的输出:                                        │
│   - 视觉描述 + 文本推理 → 综合结论                             │
│   - 发现跨模态矛盾 → 标记 Escalation                           │
│   - 计算综合置信度                                             │
└──────────────────────────┬───────────────────────────────────┘
                           ▼
┌──────────────────────────────────────────────────────────────┐
│ Step 7: Quality Estimator + Escalation Gate                   │
│   Check:                                                      │
│   - confidence >= threshold?                                  │
│   - schema_valid?                                             │
│   - tool_tests_passed?                                        │
│   - cross_modal_consistent?                                   │
│   任一失败 → Escalate (换策略/升级模型/请求人工)              │
│   全部通过 → 返回最终结果                                      │
└──────────────────────────┬───────────────────────────────────┘
                           ▼
┌──────────────────────────────────────────────────────────────┐
│ Step 8: Response Adapter                                      │
│   转换为客户端期望的格式:                                      │
│   - Claude Code ← Anthropic /v1/messages 响应 (SSE 流式)     │
│   - OpenCode  ← OpenAI /v1/chat/completions 响应 (SSE 流式)  │
│   - Codex     ← OpenAI /v1/responses 或 chat/completions     │
└──────────────────────────┬───────────────────────────────────┘
                           ▼
┌──────────────────────────────────────────────────────────────┐
│ Step 9: Log & Learn (JSONL Audit Trail — from MiroFish)      │
│   记录: {request, strategy, models_used, cost, latency,       │
│          confidence, escalated, success}                      │
│   写入 SQLite + JSONL 审计日志                                 │
│   定期从日志中学习: 更新难度分类器、优化路由阈值               │
└──────────────────────────────────────────────────────────────┘
```

### 3.2 分类器 → 路由规则映射

TaskClassifier 输出 `TaskProfile` 结构，StrategySelector 将其与 `models.yaml` 中的 `routing.rules` 匹配：

```rust
// orchestration/context/task_classifier.rs 输出
pub struct TaskProfile {
    pub task_type: TaskType,        // ← 从请求内容推断
    pub complexity: f64,            // 0.0-1.0
    pub risk: Risk,                 // Low/Medium/High/Critical
    pub verifiability: f64,         // 0.0-1.0
    pub has_image: bool,            // ← ModalityRouter 提供
    pub has_text: bool,
    pub tightly_coupled: bool,      // ← ModalityRouter 提供
}
```

**task_type 推导规则**（分类器内嵌的规则引擎）：

| 请求特征 | 推导的 task_type | 匹配的 YAML rule |
|----------|-----------------|------------------|
| 短文本，无代码，无工具 | `summary` / `rewrite` / `translate` | `strategy: route, model: cheap_executor_fast` |
| 包含图片，无文本分析 | `image_description` | `strategy: route, model: vision_expert` |
| 包含代码片段，简单修改 | `simple_code` | `strategy: route, model: cheap_executor_code` |
| 包含代码，`complexity >= 0.5` | `coding` | `strategy: cascade` |
| `complexity >= 0.7, risk >= high` | `coding` (高难) | `strategy: debate` |
| `has_image && has_text && !tightly_coupled` | 跨模态 | `strategy: cross_modal` |
| `complexity >= 0.8, risk == critical` | 极高难 | `strategy: moa` |

**匹配流程**: TaskClassifier 填充 `TaskProfile` → StrategySelector 遍历 `routing.rules[]` 按优先级匹配 → 首条匹配的规则决定策略和模型。无规则匹配时，回退到 `routing.default_strategy: route`。

### 3.3 五种核心策略详解

#### 策略 1: ROUTE（智能路由）

```
输入 → Difficulty Classifier → 单选最佳模型 → 直接返回

规则:
  summary/rewrite/translate  → cheapest_text (DeepSeek/GLM/Qwen)
  image_description          → vision_expert (Qwen-VL/Gemini-Vision)
  speech_transcribe          → whisper_local
  simple_code                → cheap_coder + lint + test
  code_with_tests            → mid_coder + test_loop
  complex_architecture       → frontier_planner (Claude Opus/GPT-5)

成本: 极低 (只调用一个便宜模型)
延迟: 极低 (单次调用)
适用: 60-70% 的日常请求
```

#### 策略 2: CASCADE（级联升级）

```
便宜模型 → 质量估计 → 通过? → 返回
                    → 不通过 → 中端模型 → 质量估计 → 通过? → 返回
                                                  → 不通过 → 顶级模型

升级触发条件:
  - confidence < 0.65
  - schema 校验失败
  - 测试失败 (代码场景)
  - 自评不确定性 > 阈值
  - 工具验证报错

成本: 低 (80% 请求在第一级完成)
延迟: 低-中 (升级时增加 2-3 秒)
适用: 20-25% 的中等复杂度请求
```

#### 策略 3: CROSS-MODAL（跨模态编排）★ 核心差异化策略

```
多模态请求
    │
    ▼
Modality Decomposer (判断模态可分离性)
    │
    ├── 视觉路径: 专用视觉模型 → 结构化描述
    ├── 语音路径: ASR → 文本转录
    ├── 文本路径: 上下文推理
    └── 工具路径: 代码执行/文件操作
    │
    ▼
Cross-Modal Aggregator (融合多路径结果)
    │
    ├── 一致性检查: 视觉描述和文本推理是否矛盾?
    ├── 信息完整性: 所有模态路径是否覆盖了任务需求?
    └── 置信度计算: 综合各路径置信度
    │
    ▼
一致且高置信 → 融合输出
矛盾或低置信 → 升级到原生多模态强模型 (GPT-5/Claude Opus)
```

#### 策略 4: DEBATE（多模型辩论）

```
同一个问题，3个不同模型独立回答
    │
    ▼
Judge 模型比较答案:
  - 多数一致 → 采纳多数
  - 两两矛盾 → 强 Judge 裁决
  - 完全不一致 → 升级到最强模型
    │
    ▼
工具验证 (如有) → 最终答案

适用: 高风险事实核查、代码审查、安全审计、法律/合规
成本: 高 (3-5 次模型调用)
延迟: 高 (并行可缓解)
```

#### 策略 5: MoA（混合智能体聚合）

```
Layer 1: N 个 Proposer 模型独立生成候选方案
    │
Layer 2: Aggregator 读取所有 Proposer 输出 → 生成改进方案
    │
Layer 3: (可选) 第二个 Aggregator 进一步精炼
    │
工具验证 + 最终审查

Proposer 选择原则:
  - 3-5 个能力相近但异构的模型
  - 全都通过质量底线（排除太弱的模型）
  - 多样性来源: 不同厂商、不同架构、不同训练数据

适用: 复杂架构设计、产品战略、高价值方案、多语言文档
成本: 最高
延迟: 最高
```

## 4. MiroFish 模式移植详解

> 本节记录从 MiroFish (Python/Flask) 移植到 OmniAgent (Rust/Tauri) 的架构模式。
> 这些模式经过 MiroFish 生产验证，在 OmniAgent 中以 Rust 惯用法重新实现。

### 4.1 移植映射表

| MiroFish 组件 | OmniAgent Rust 模块 | 移植要点 |
|---|---|---|
| `SimulationManager` 6态生命周期 | `orchestration/workflow/lifecycle.rs` | Enum `WorkflowStatus` + SQLite 持久化 |
| `ReportAgent` ReACT 循环 | `orchestration/strategy/react_executor.rs` | 泛型 `Tool` trait + `ReactLoop<S>` |
| `OntologyGenerator` | `orchestration/context/knowledge_graph.rs` | LLM 返回 `OntologySchema` serde 结构体 |
| `ReportLogger` JSONL 审计 | `orchestration/audit/jsonl_logger.rs` | 异步 JSONL + tracing 集成 |
| `ReportManager` 章节输出 | `orchestration/workflow/section_writer.rs` | tokio::fs 流式输出 + 增量进度 |
| `SimulationConfigGenerator` | `orchestration/strategy/config_advisor.rs` | LLM 配置调优 + 成本/质量评估集成 |
| File-based state persistence | SQLite via `sqlx` | JSON 文件状态 → 类型化 SQLite 查询 |

### 4.2 不移植的组件

| MiroFish 组件 | 原因 |
|---|---|
| OASIS simulation integration | 社交模拟专有，与多模型编排无关 |
| Zep Cloud 知识图谱服务 | 替换为本地 SQLite + Qdrant |
| MBTI/personality generation | 领域特定，不属于编排关注点 |
| Twitter/Reddit platform scripts | MiroFish 用例专有 |
| Vue 3 frontend | OmniAgent 使用 React + Tauri IPC |
| Flask HTTP server | OmniAgent 使用 axum (Rust async HTTP) |

### 4.3 生命周期状态机 (from SimulationManager)

MiroFish 的 `SimulationManager` 实现了一个健壮的状态机，具有文件持久化功能。在 OmniAgent 中，这成为了工作流执行的基础：

```rust
// orchestration/workflow/lifecycle.rs

#[derive(Debug, Clone, Serialize, Deserialize, sqlx::Type, PartialEq)]
#[serde(rename_all = "snake_case")]
pub enum WorkflowStatus {
    Created,
    Preparing,
    Ready,
    Running,
    Paused,
    Completed,
    Failed,
}

#[derive(Debug, Serialize, Deserialize)]
pub struct WorkflowState {
    pub workflow_id: String,
    pub status: WorkflowStatus,
    pub current_step: Option<String>,
    pub completed_steps: Vec<String>,
    pub error: Option<String>,
    pub created_at: chrono::DateTime<chrono::Utc>,
    pub updated_at: chrono::DateTime<chrono::Utc>,
}

impl WorkflowState {
    pub fn transition(&mut self, new_status: WorkflowStatus) -> Result<(), WorkflowError> {
        // Enforce valid transitions (from MiroFish's implicit state machine)
        match (&self.status, &new_status) {
            (Created, Preparing) | (Created, Failed)
            | (Preparing, Ready) | (Preparing, Failed)
            | (Ready, Running) | (Ready, Failed)
            => {},
            (Running, Completed) | (Running, Failed) | (Running, Paused) => {},
            (Paused, Running) | (Paused, Failed) => {},
            _ => return Err(WorkflowError::InvalidTransition {
                from: self.status.clone(),
                to: new_status,
            }),
        }
        self.status = new_status;
        self.updated_at = chrono::Utc::now();
        Ok(())
    }
}
```

### 4.4 ReACT 执行器 (from ReportAgent)

MiroFish 的 `ReportAgent` 实现了一个生产级 ReACT 循环，包含：
- 最小工具调用强制执行
- 迭代限制
- 工具多样性跟踪
- 冲突解决（同时包含工具调用和最终答案时）
- 强制终止

在 OmniAgent 中，这成为了 DEBATE/MoA/CROSS-MODAL 策略的通用执行原语：

```rust
// orchestration/strategy/react_executor.rs

#[async_trait]
pub trait Tool: Send + Sync {
    fn name(&self) -> &str;
    fn description(&self) -> &str;
    fn parameters_schema(&self) -> serde_json::Value;
    async fn execute(&self, params: serde_json::Value) -> Result<ToolOutput>;
}

pub struct ReactExecutor {
    tools: HashMap<String, Box<dyn Tool>>,
    llm_client: LlmClient,
    max_iterations: usize,      // Default: 5
    min_tool_calls: usize,      // Default: 3 (from MiroFish)
    jsonl_logger: JsonlLogger,  // Structured audit (from MiroFish)
}

pub enum ReactOutput {
    ToolCall { name: String, params: serde_json::Value },
    FinalAnswer { content: String },
}

impl ReactExecutor {
    pub async fn execute(&self, system_prompt: &str, user_prompt: &str) -> Result<String> {
        let mut messages = vec![
            Message::system(system_prompt),
            Message::user(user_prompt),
        ];
        let mut tool_calls_count = 0;
        let mut used_tools = HashSet::new();
        let all_tools: HashSet<String> = self.tools.keys().cloned().collect();

        for iteration in 0..self.max_iterations {
            let response = self.llm_client.chat(&messages).await?;
            let parsed = self.parse_response(&response)?;

            match parsed {
                ReactOutput::ToolCall { name, params } => {
                    // Check iteration limit (from MiroFish)
                    if tool_calls_count >= self.max_iterations {
                        self.force_final_answer(&mut messages).await?;
                        break;
                    }

                    let result = self.execute_tool(&name, params).await?;
                    tool_calls_count += 1;
                    used_tools.insert(name.clone());

                    // Log to JSONL (from MiroFish ReportLogger)
                    self.jsonl_logger.log_tool_call(&name, tool_calls_count, iteration).await;

                    // Inject observation (MiroFish pattern)
                    messages.push(Message::assistant(&response));
                    messages.push(Message::user(&format_observation(&name, &result)));
                }
                ReactOutput::FinalAnswer { content } => {
                    // Enforce min tool calls (from MiroFish)
                    if tool_calls_count < self.min_tool_calls {
                        let unused = all_tools.difference(&used_tools);
                        messages.push(Message::assistant(&response));
                        messages.push(Message::user(&format_insufficient_tools(
                            tool_calls_count, self.min_tool_calls, unused
                        )));
                        continue;
                    }
                    self.jsonl_logger.log_final_answer(&content, tool_calls_count).await;
                    return Ok(content);
                }
            }
        }

        // Force final answer at max iterations (from MiroFish)
        self.force_final_answer(&mut messages).await
    }
}
```

### 4.7 增量章节写入器 (from ReportManager)

MiroFish 逐节写入报告部分，带有进度跟踪。OmniAgent 将此移植用于工作流输出流式传输：

```rust
// orchestration/workflow/section_writer.rs

pub struct SectionWriter {
    output_dir: PathBuf,
}

impl SectionWriter {
    pub async fn save_section(&self, index: usize, title: &str, content: &str) -> Result<()> {
        let filename = format!("section_{:02}.md", index);
        let path = self.output_dir.join(&filename);
        let md = format!("## {}\n\n{}\n\n", title, content);
        tokio::fs::write(&path, md).await?;
        self.update_progress(index, title).await?;
        Ok(())
    }

    pub async fn update_progress(&self, current: usize, current_title: &str) -> Result<()> {
        let progress = serde_json::json!({
            "current_section": current_title,
            "updated_at": chrono::Utc::now().to_rfc3339(),
        });
        let path = self.output_dir.join("progress.json");
        tokio::fs::write(&path, serde_json::to_string_pretty(&progress)?).await?;
        Ok(())
    }

    pub async fn assemble_full_output(&self, title: &str, summary: &str) -> Result<String> {
        // Read all section_*.md files in order and concatenate
        // (from MiroFish's assemble_full_report pattern)
        let mut output = format!("# {}\n\n> {}\n\n---\n\n", title, summary);
        let mut entries = tokio::fs::read_dir(&self.output_dir).await?;
        let mut sections: Vec<(usize, String)> = vec![];
        while let Some(entry) = entries.next_entry().await? {
            let name = entry.file_name().to_string_lossy().to_string();
            if name.starts_with("section_") && name.ends_with(".md") {
                let num: usize = name.replace("section_", "").replace(".md", "").parse()?;
                let content = tokio::fs::read_to_string(entry.path()).await?;
                sections.push((num, content));
            }
        }
        sections.sort_by_key(|(n, _)| *n);
        for (_, content) in sections {
            output.push_str(&content);
        }
        let full_path = self.output_dir.join("full_output.md");
        tokio::fs::write(&full_path, &output).await?;
        Ok(output)
    }
}
```

## 5. 跨模态编排 — 关键算法

### 5.1 Modality Decomposer 算法

Phase 1-2 使用显式规则表（非公式）：

```yaml
decomposition_rules:
  - has_image AND has_text → SPLIT: vision→description→text_reasoning
  - has_audio AND has_text → SPLIT: speech→transcription→text_processing
  - has_image AND has_text AND tightly_coupled_context → NO_SPLIT: route to multimodal_frontier
  - has_audio AND has_image AND has_text → SPLIT_ALL: speech→text, vision→text, then merge
```

Phase 5 引入评分公式（保留为 ML 优化目标）：

```
DecomposabilityScore(task) =
  0.25 × ModalityIndependence
+ 0.25 × VerificationAvailable
+ 0.20 × StepLossTolerance
+ 0.15 × LatencyBudget
+ 0.15 × ErrorRecoverability

Score >= 0.70 → 可拆分
Score 0.40-0.70 → 部分可拆分
Score < 0.40 → 不可拆分
```

### 5.2 跨模态信息损失最小化

```
方案 1: 结构化描述协议
  视觉模型输出必须包含:
  {
    "objects": [{"label": "IC芯片", "bbox": [x1,y1,x2,y2], "attributes": {...}}],

    "uncertain_areas": [{"bbox": [...], "reason": "反光看不清丝印"}]
  }

方案 2: 多轮交互
  推理模型可以追问: "你提到的电容C1旁边的IC是什么型号？"
  视觉模型: 裁切该区域重新识别

方案 3: 多视觉模型交叉验证
  两个不同视觉模型独立描述 → 比较 → 不一致部分标记为 uncertain
```


## 14. 关键设计决策与避坑指南

### 14.1 架构决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 编排引擎语言 | 纯 Rust (核心路径) | 零成本抽象、内存安全、单一二进制 |
| 工作流定义 | YAML 声明式 + 运行时动态修改 | 可配置、可热更新 |
| 协议兼容 | Anthropic Messages + OpenAI Chat Completions | 覆盖所有主流客户端 |
| 向量存储 | Qdrant 本地模式 | 无需外部服务 |
| UI 框架 | Tauri 2.x | CC-Switch 参考、跨平台 |
| MiroFish 集成 | 模式移植到 Rust (非代码引用) | 单一运行时、无 Python 依赖 |

### 14.2 避坑清单

**坑 1: 质量估计器是系统最薄弱环节**
- Phase 1-2 仅用二元工具验证，不用模型自评置信度
- Phase 5 积累数据后引入连续评分

**坑 2: 跨模态信息损失**
- 视觉描述必须结构化（带坐标、置信度、不确定区域标记）
- 关键区域必须裁切放大二次分析
- 推理模型必须能回问视觉模型

**坑 3: 模型多样性 vs 质量底线**
- 弱模型会拉低聚合结果
- Proposer 必须过质量门槛
- 同厂商模型的错误模式可能高度相关

**坑 4: 延迟爆炸**
- Cascade 首模型 streaming 立即透传
- 交互式场景用 ROUTE，批处理用 Cascade/MoA
- 无依赖节点必须并行执行

**坑 5: 上下文污染**
- 每个 Worker 只接收必要上下文
- 多 Agent 输出经过 Aggregator 筛选
- 错误输出必须被标记

**坑 6: 评测集建设**
- 必须有自己的 Golden Task Set
- 每次换模型/改策略都跑一次评测

**坑 7: 成本幻觉**
- 多模型调用可能比单强模型更贵
- 必须设每日预算上限
- Cost Ledger 要实时可见

## 15. 架构差距修复 (Cross-Model Audit)

### D1: Streaming 首 token 直传

所有策略支持首 token 直传：
- ROUTE: 直接透传
- CASCADE: 第一级模型 streaming 立即透传，后台异步运行第二级
- CROSS-MODAL: 首先完成的模态路径立即 streaming
- DEBATE/MoA: 显示"综合中..."loading，结果合并后 streaming 输出

### D2: 质量估算器 Phase 1 仅用工具验证

Phase 1-2: `QualityScore = ToolVerification.passed AND SchemaCompliance.valid`
Phase 5: 引入连续评分

### D3: 模态分解 Phase 1 用规则

显式规则表替代公式，Phase 5 引入 ML 评分

### D4: 成本数据增加更新时间戳

每个 model entry 增加 `cost_last_updated` 字段，新增 `/v1/models/costs` API

### D5: LiteLLM 集成 HTTP Sidecar

Phase 1-2 纯 Rust 直调 Provider API，Phase 3+ 可选 LiteLLM HTTP Sidecar

### D6: 取消语义

DAG 节点增加 `on_cancel` 钩子，返回 partial 结果

### D7: 工具验证增加可质疑标记

`tool_confidence: 1.0 | 0.8 | 0.5`，0.5 级结果可被模型质疑

### D8: 升级阈值 Phase 1 二元判断

Phase 1-4: 二元判断（工具测试 pass/fail、Schema valid/invalid）
Phase 5: A/B 测试确定连续阈值
