## 一句话描述

**OmniAgent Workbench** 是一个 Fork 自 CC-Switch 的本地桌面应用，在 CC-Switch 的 Provider 管理 + 代理转发基础之上，新增 MiroFish 验证模式的编排引擎，让 Claude Code / Codex / OpenCode 等 AI 编程客户端的每个回答都经过**工具验证 + 多模型交叉检查 + 自动升级**，答案质量超越任何单一前沿模型。

---

## 问题定义

### 现状痛点

开发者使用 Claude Code / Codex 等 AI 编程工具时：
1. **答案不可靠** — 单一模型偶尔产出幻觉代码，花更多时间审查和修复
2. **无法自动验证** — 生成的代码没有经过测试运行、lint 检查、schema 校验
3. **成本不透明** — 各自调用 API，没有统一的成本追踪和预算控制
4. **策略不可调** — 现有网关只做转发，无法定义「代码任务用 3 个模型辩论」这样的策略

### MiroFish 已验证的解决方案

MiroFish 的 ReACT 循环已在社交模拟预测中证明了核心模式：
- **最少 3 次工具调用** 强制执行 — 阻止 LLM 走捷径（v2 适配：异构验证步骤）
- **工具多样性跟踪** — 确保不重复使用同一个信息源
- **冲突解决** — 当 LLM 同时产出工具调用和最终答案时，优雅降级
- **温度衰减重试** — 每次重试降低温度 0.1，收敛到稳定输出
- **JSON 自我修复** — 多级解析策略，应对 LLM 格式不规整

## MiroFish 核心模式移植

### 1. ReACT 循环 (from report_agent.py)

**MiroFish 原始实现：**
- 最少 3 次工具调用强制执行（lines 1377-1389）
- 工具多样性跟踪，提示未使用的工具（lines 1448-1454）
- 冲突解决：同时出现工具调用和 Final Answer 时，先重试 2 次，第 3 次降级（lines 1329-1361）
- 最大 5 轮迭代，超出后强制终止
- 多格式解析：XML 标签 > 裸 JSON 兜底（lines 1067-1125）

**OmniAgent Rust 移植：**

```rust
// orchestration/react_executor.rs

pub struct ReACTExecutor {
    max_iterations: u32,            // 默认 5
    min_verification_steps: u32,    // 默认 3 — 至少 3 个异构验证步骤
    max_conflict_retries: u32,      // 默认 2
    all_tools: HashSet<String>,
}

pub struct ReACTResult {
    pub answer: String,
    pub tool_calls_made: u32,
    pub tools_used: HashSet<String>,
    pub iterations: u32,
    pub quality_passed: bool,
}

impl ReACTExecutor {
    pub async fn execute(&self, task: TaskProfile) -> Result<ReACTResult> {
        let mut tool_calls_count = 0;
        let mut used_tools = HashSet::new();
        let mut conflict_retries = 0;

        for iteration in 0..self.max_iterations {
            let response = self.call_model(task.clone()).await?;

            // MiroFish 模式: 检测冲突 (tool_call + Final Answer 同时出现)
            if response.has_tool_call() && response.has_final_answer() {
                conflict_retries += 1;
                if conflict_retries >= self.max_conflict_retries {
                    // 第 3 次冲突: 强制降级，只取工具调用
                    return self.force_tool_only(response);
                }
                continue; // 重试
            }

            // 最少验证步骤强制（MiroFish 模式适配：v2 为异构验证步骤，非外部工具调用）
            if response.has_final_answer() && tool_calls_count < self.min_verification_steps {
                let unused = self.all_tools.difference(&used_tools);
                // 提示使用未使用的验证工具
                self.inject_tool_hint(unused);
                continue;
            }

            if response.has_tool_call() {
                tool_calls_count += 1;
                used_tools.insert(response.tool_name().clone());
                // MiroFish 模式: 跟踪工具多样性
                self.execute_tool(response.tool_call()).await?;
            }

            if response.has_final_answer() {
                return Ok(ReACTResult {
                    answer: response.final_answer(),
                    tool_calls_made: tool_calls_count,
                    tools_used,
                    iterations: iteration + 1,
                    quality_passed: false, // 后续由 QualityGate 判定
                });
            }
        }

        // MiroFish 模式: 强制终止
        self.force_termination()
    }
}
```

### 2. 质量门控 (from report_agent.py 验证模式)

```rust
// orchestration/quality_gate.rs

pub enum VerificationTool {
    TestRunner,      // 运行单元测试
    Linter,          // eslint / clippy / ruff
    SchemaValidator, // JSON schema 校验
    CompileChecker,  // 编译检查
    LLMCrossCheck,   // LLM 交叉验证
}

pub struct QualityGate {
    tools: Vec<VerificationTool>,
    pass_threshold: f64,  // 默认 0.65 (OmniAgent 设计选择，非 MiroFish 源码)
}

impl QualityGate {
    pub async fn verify(&self, response: &str, task: &TaskProfile) -> QualityResult {
        let mut scores = Vec::new();

        for tool in &self.tools {
            let score = match tool {
                VerificationTool::TestRunner => self.run_tests(response).await?,
                VerificationTool::Linter => self.run_linter(response).await?,
                VerificationTool::SchemaValidator => self.validate_schema(response).await?,
                VerificationTool::CompileChecker => self.compile_check(response).await?,
                VerificationTool::LLMCrossCheck => self.llm_cross_check(response).await?,
            };
            scores.push(score);
        }

        let avg = scores.iter().sum::<f64>() / scores.len() as f64;
        QualityResult {
            passed: avg >= self.pass_threshold,
            score: avg,
            individual_scores: scores,
        }
    }
}
```

// orchestration/retry_policy.rs

pub struct TemperatureDecayRetry {
    base_temperature: f64,    // 默认 0.7
    decay_per_attempt: f64,   // 默认 0.1 (from MiroFish)
    max_attempts: u32,        // 默认 3
}

impl TemperatureDecayRetry {
    pub fn temperature_for_attempt(&self, attempt: u32) -> f64 {
        (self.base_temperature - (attempt as f64 * self.decay_per_attempt))
            .max(0.1) // 最低温度 0.1
    }
}

### 4. JSON 自我修复 (from simulation_config_generator.py lines 483-533)

```rust
// orchestration/json_healer.rs

pub fn heal_json(raw: &str) -> Result<Value> {
    // Level 1: 直接解析
    if let Ok(v) = serde_json::from_str(raw) {
        return Ok(v);
    }

    // Level 2: 关闭未闭合的括号 (from MiroFish _fix_truncated_json)
    let healed = close_unclosed_brackets(raw);
    if let Ok(v) = serde_json::from_str(&healed) {
        return Ok(v);
    }

    // Level 3: 提取 JSON 块 (from MiroFish _try_fix_config_json)
    let extracted = extract_json_block(raw);
    if let Ok(v) = serde_json::from_str(&extracted) {
        return Ok(v);
    }

    // Level 4: 移除控制字符 + 正则化空白
    let cleaned = clean_control_chars(&extracted);
    serde_json::from_str(&cleaned)
        .map_err(|e| anyhow!("JSON healing failed after 4 levels: {}", e))
}
```

### 热重载机制

```rust
// orchestration/strategy_loader.rs

pub struct StrategyLoader {
    path: PathBuf,
    last_modified: SystemTime,
    strategies: Arc<RwLock<StrategyConfig>>,
}

impl StrategyLoader {
    pub async fn watch(&self) {
        let mut interval = tokio::time::interval(Duration::from_secs(5));
        loop {
            interval.tick().await;
            let modified = fs::metadata(&self.path)
                .ok()
                .and_then(|m| m.modified().ok());

            if modified != Some(self.last_modified) {
                match self.reload() {
                    Ok(new_config) => {
                        *self.strategies.write().await = new_config;
                        self.last_modified = modified.unwrap();
                        // 通知 UI 策略已更新
                        self.notify_ui_reload();
                    }
                    Err(e) => {
                        // 加载失败，保持旧配置，通知 UI 错误
                        self.notify_ui_error(e);
                    }
                }
            }
        }
    }
}
```

## 关键设计决策

### D1: 为什么答案质量比速度/成本更重要

开发者的痛点不是「AI 太贵」或「AI 太慢」，而是「AI 给的代码有时候是错的」。一个错误的代码提交可能花 2 小时 debug，远超省下的 API 费用。所以核心目标是**答案质量**。

### D2: 为什么 YAML + UI 双通道

不同用户偏好不同：
- 高级工程师：愿意改 YAML，需要 git diff 和版本控制
- 技术主管：想看可视化流程图，拖拽调参数
- 其他协作者：只看统计和成本，不碰配置

单通道只服务一种人。双通道服务所有人。

### D3: 为什么 Fork CC-Switch 而非全新项目

CC-Switch 已经有：
- 22+ shadcn/ui 组件（Button、Dialog、Card、Tabs、Switch 等）
- 完整的 Provider 管理 CRUD
- Axum 代理服务器（支持 SSE 流、故障转移、断路器）
- 使用统计仪表盘（recharts 图表）
- 国际化（i18next）
- 自动更新
- 暗色/亮色主题

从零做这些需要 6-8 周。Fork 后只需加编排引擎。

### D4: 为什么不是 Python 混合方案

用户不想安装 Python + Node + Rust。一个 Tauri 桌面应用 = 一个安装包。MiroFish 的模式是**移植为 Rust**，不是运行 Python 代码。

## 设计决策补充（审阅后修复）

### SSE 流式 UX 决策

采用**方案 C: ROUTE 流式、非 ROUTE 显示加载状态**。

- **ROUTE 策略**：直接流式透传，用户看到实时 token 输出（与直连模型体验一致）
- **CASCADE / DEBATE / MoA**：客户端显示加载指示器（"正在编排中..."），编排完成后一次性发送完整响应。原因：CASCADE 需要等第一级模型完成 + 验证通过后才能决定是否升级，无法做到真正的流式直通
- **CROSS-MODAL**：每个模态路径独立完成后合并，也是一次性发送

这对 Claude Code 的体验影响是：CASCADE 任务会有几秒到几十秒的等待，然后收到完整回答。作为补偿，UI 仪表盘实时显示编排进度（"第 1 级模型运行中..."、"质量验证中..."、"升级到第 2 级..."）。

### TaskClassifier 规格

**输入**：解析后的 Anthropic/OpenAI 请求体（messages 数组 + tools 数组 + model 字段）

**算法**：规则引擎（Phase 1），不使用 LLM 调用。基于以下启发式：

| 字段 | 推导规则 | 输出 |
|------|---------|------|
| `task_type` | 消息中包含代码块 (`\`\`\``) → `coding`；图片 content → `image`；短消息且无工具 → `summary` | enum |
| `complexity` | 消息总 token 数 / 4000（归一化到 0-1）；包含 tool_use → +0.3；多轮对话 → +0.2 | 0.0-1.0 |
| `risk` | 包含 "delete/remove/drop/truncate" → `critical`；包含 "refactor/migrate" → `high`；包含 "fix/update" → `medium`；其他 → `low` | enum |
| `verifiability` | 有 tool_use 且 tool 可自动运行 → 0.9；有 tool_use 但不可自动运行 → 0.5；无 tool → 0.1 | 0.0-1.0 |
| `has_image` | 消息中包含 image_url 或 image content block | bool |
| `need_code` | 消息中包含代码块或编程关键词（"implement/write/create function"） | bool |

**输出**：`TaskProfile` 结构体（YAML 策略匹配使用全部字段）

### 工具验证环境（收窄范围）

v2 MVP 的工具验证**不运行外部命令**（cargo test、eslint 等）。原因：网关没有用户项目目录的上下文，代码片段脱离项目无法运行有意义的测试。

**v2 验证方式**：
1. **结构化检查**：解析 LLM 输出的代码块，检查语法完整性（括号匹配、import 存在、函数签名闭合）
2. **LLM 自评**：用 judge 模型评估代码质量（0-1 分）
3. **模式匹配**：检查常见错误模式（空 catch、未处理异常、硬编码路径等）
4. **Schema 校验**：JSON 输出用 JSON Schema 验证

**v3+ 增强**：接受用户配置项目路径，在沙箱子进程中运行真实测试。这需要 IDE 集成或 CLI hook。

### 企业特性分阶段

**v2 (MVP)**：单用户桌面应用，不含团队管理。包含：预算追踪、成本仪表盘、策略配置。
**v5+ (Server 部署)**：多用户团队、RBAC、团队预算、API Key 共享。

### 热重载改进

使用 `notify` crate 的文件系统监听替代 5 秒轮询，加 2 秒去抖（等文件写入稳定后再重载）。

### YAML-UI 同步语义

**Single source of truth: YAML 文件。**

- YAML 文件是策略的唯一权威来源（磁盘即状态）
- UI 编辑器读写 YAML 文件：保存时写入磁盘，`notify` 触发热重载
- UI 打开时从 YAML 加载当前状态，不从内存缓存读取
- 冲突处理：如果 UI 编辑期间 YAML 被外部修改，UI 显示冲突提示并提供 diff/merge 选项

### UI 状态定义

编排面板的每个策略卡片需要处理以下状态：

| 状态 | 触发条件 | UI 表现 |
|------|---------|---------|
| **Idle** | 无活跃编排请求 | 显示策略参数和上次统计 |
| **Loading** | 编排请求进行中 | 进度指示器 + "正在编排中..." + 当前阶段文本 |
| **Active** | 编排请求正在执行策略 | 实时显示当前级别模型、验证结果 |
| **Success** | 编排完成，响应已返回 | 绿色勾 + 最终模型 + 耗时 + 成本 |
| **Error** | 编排过程中出错（模型不可用、预算超限等） | 红色错误信息 + 重试按钮 + 降级提示 |
| **Partial** | 部分验证通过但未达阈值，已升级到最终模型 | 黄色警告 + 升级次数 + 各级验证结果 |
| **Empty** | 无编排历史记录 | 引导用户开启编排 + 配置策略 |

### 边界条件

#### QualityGate 空工具列表

当 `verification.tools` 为空时，QualityGate 的 `avg` 计算会除以零（`inf >= 0.65` 恒为 true）。

```rust
// 修复：YAML 加载时验证
impl StrategyConfig {
    pub fn validate(&self) -> Result<()> {
        if self.verification.tools.is_empty() {
            anyhow::bail!("verification.tools 不能为空 — 至少需要一个验证工具");
        }
        Ok(())
    }
}
```

#### 预算竞态条件

多个并发请求同时通过预算检查，在任何一个完成扣费前都认为预算充足。

**修复：两阶段预算**

```rust
// 请求开始时：估算扣费（冻结额度）
let reserved = budget.reserve_estimated(task).await?;
// 请求完成时：按实际用量结算
budget.settle(reserved, actual_cost).await?;
```

`reserve_estimated` 基于历史数据估算本次请求成本（如按 token 数 × 模型单价），冻结该额度。多个并发请求共享同一预算池，冻结额度累加，超出限制则拒绝新请求。

#### 热重载中途配置切换

请求进行中的 CASCADE 升级可能因热重载导致策略配置变更（如升级目标模型被移除）。

**修复：请求持有配置快照**

```rust
// 每个请求开始时持有 Arc<StrategyConfig> 快照
let config_snapshot = self.config.read().await.clone();
// 请求全程使用快照，不受热重载影响
executor.execute(task, config_snapshot).await?;
// 热重载更新全局 config，但已有请求继续使用旧快照
```
