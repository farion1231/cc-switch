# 2.7 Codex 记忆/Chronicle 功能

> 适用版本：CC Switch v3.16.5+（参考 [OpenAI Codex 记忆文档](https://developers.openai.com/codex/memories)）

Codex CLI 内置了「记忆」与「Chronicle」功能，会在每个会话结束后自动抽取要点、跨会话整合。`~/.codex/config.toml` 中通过 `[memories]` 段控制：

```toml
[memories]
generate_memories = true                       # bool：是否启用记忆生成
use_memories = true                            # bool：是否在新会话中注入已有记忆
disable_on_external_context = false            # bool：外部上下文（MCP / Web 搜索）触发的会话是否参与记忆生成
min_rate_limit_remaining_percent = 25          # int：剩余速率限制百分比阈值
extract_model = "MiniMax-M3"             # 用于逐线程记忆抽取的模型
consolidation_model = "gpt-5.4-mini"      # 用于跨线程记忆整合的模型
```

`extract_model` 与 `consolidation_model` 显式配置时，Codex 后端会调用这两个模型执行记忆处理。**若未配置，则回退到顶层 `model`**（即你当前供应商的主模型）。

> ⚠️ **关键前提**：Codex 的记忆/Chronicle 功能是受 **Feature Flag** 控制的。即使配置了 `[memories]` 段，Codex 仍要求 `config.toml` 中显式启用：
>
> ```toml
> [features]
> memories = true
> ```
>
> CC Switch v3.16.5+ 的「启用 Codex 记忆功能」开关**会自动写入该 feature flag**——你无需手动在 `config.toml` 中追加这段。但要注意，Codex 还有其他未公开的 feature flag 需要时也放在这里（例如 `web_search_request`）。

## 常见问题：自定义供应商下记忆功能报错

### 现象

切到第三方/自定义供应商后，Codex 启动时报类似错误：

```
Codex memory/chronicle backend failed: model "MiniMax-M3" not found
```

或 `extract_model` / `consolidation_model` 报「模型不存在」。

### 原因

`[memories]` 段里硬编码的 `extract_model` / `consolidation_model` 指向的是某个**特定模型**（例如 `MiniMax-M3`、`gpt-5.4-mini`）。当你切换到一个**不提供该模型**的自定义供应商时，Codex 后端会因找不到模型而完全跳过记忆抽取/整合，记忆功能失效。

### 解决：使用「启用 Codex 记忆功能」开关

CC Switch v3.16.5+ 在 Codex 配置面板的 **config.toml (TOML) 编辑器**行新增了一个「**启用 Codex 记忆功能**」复选框，位于「启用 Goal mode / 启用远程压缩 / 写入通用配置」三个复选框的**最左侧**。

**勾选后**，cc-switch 会同步管理三处：

- `[memories].extract_model` 与 `[memories].consolidation_model`：写入当前供应商的 `model`（在「模型映射 / 模型目录」中维护）。切换供应商后，下次保存会自动同步新供应商的 model。
- `[features].memories = true`：Codex 真正启用记忆功能的 feature flag。**此字段由开关自动写入**，无需手动追加。
- 切换供应商时上面三处都会跟着新 model 重写。

**取消勾选后**，cc-switch 只移除这三处键/段：

- `[memories]` 段内的 `extract_model` / `consolidation_model` 被删除，其他字段（`generate_memories` / `use_memories` / `disable_on_external_context` / `min_rate_limit_remaining_percent`）原样保留
- `[features].memories` 被删除，段内其他字段（如 `web_search`）原样保留
- 若 `[memories]` 段因删除后只剩空壳，则整段一并清理（反之亦然）
- Codex 走默认回退路径（用顶层 `model`）

### 操作步骤

1. 打开要编辑的 Codex 供应商卡片，点击「编辑」
2. 确认「模型映射 / 模型目录」第一行已填入 `model`（cc-switch 同步 `extract_model` / `consolidation_model` 时会读这个值）
3. 勾选「**启用 Codex 记忆功能**」复选框
4. 点击「保存」
5. 重启 Codex CLI（按 Code 官方要求刷新模型列表与记忆配置）

### 适用场景

| 场景 | 是否需要勾选 |
|------|--------------|
| 使用 OpenAI 官方供应商 | 可选（Codex 默认回退到 `model`，显式勾选更稳定） |
| 使用第三方网关（DeepSeek / GLM / Qwen / 自建 OpenAI 兼容） | **推荐勾选**，避免模型名不匹配 |
| 「模型映射」配置了多个不同名字的模型 | **推荐勾选**，记忆模型跟随主模型自动切换 |
| 想要记忆用更便宜的模型而非主模型 | **不勾选**，手动在 `config.toml` 里写固定模型名即可 |

### 常见误解

- **勾选后无法填入自定义的 extract_model？** 这是设计上的简化：让记忆用与聊天同一模型，避免跨供应商不可用。每次保存时 cc-switch 会按「模型映射 / 模型目录」第一行的 `model` 同步 `[memories].extract_model` / `[memories].consolidation_model`，**会覆盖你在 TOML 编辑器里手工改的值**。若确实需要混合配置（记忆用更便宜/不同的模型），请取消勾选，然后直接编辑 `config.toml` 里的两个 model 字段后保存。
- **修改后 Codex 没生效？** 重启 Codex CLI——Codex 启动时才会读 `config.toml`，热更新不生效。
- **「记忆」和「Chronicle」是一回事吗？** Chronicle 是 Codex 的「跨会话叙事」功能，复用同一个 `[memories].consolidation_model` 字段。
- **取消勾选后我自己配的 `generate_memories = true` 没了？** 不会。取消勾选只删 `[memories].extract_model` / `[memories].consolidation_model` 两个 model 字段以及 `[features].memories`，其他字段（如 `generate_memories` / `use_memories` / `web_search` 等）原样保留。
- **怎么临时换便宜的模型跑记忆？** 取消勾选开关，再在 `config.toml` 编辑器里把两个 model 字段改成想要的固定模型名后保存。
