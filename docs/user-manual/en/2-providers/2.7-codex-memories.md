# 2.7 Codex Memories / Chronicle

> Available in CC Switch v3.16.5+ (see [OpenAI Codex memories docs](https://developers.openai.com/codex/memories))

Codex CLI ships a built-in **Memories** and **Chronicle** feature that auto-extracts key points from each session and consolidates them across sessions. The `~/.codex/config.toml` `[memories]` table controls this:

```toml
[memories]
generate_memories = true                       # bool: enable memory generation
use_memories = true                            # bool: inject existing memories into new sessions
disable_on_external_context = false            # bool: skip sessions that used MCP / web search / tool search
min_rate_limit_remaining_percent = 25          # int: minimum remaining Codex rate-limit percentage
extract_model = "MiniMax-M3"             # model used for per-thread memory extraction
consolidation_model = "gpt-5.4-mini"      # model used for global memory consolidation
```

When `extract_model` and `consolidation_model` are set explicitly, Codex calls these models for memory work. **If unset, Codex falls back to the top-level `model`** (the active provider's main model).

## Common issue: memories fail on custom providers

### Symptom

After switching to a third-party / custom provider, Codex errors out at startup with messages like:

```
Codex memory/chronicle backend failed: model "MiniMax-M3" not found
```

Or `extract_model` / `consolidation_model` complains the model is unavailable.

### Cause

The `[memories]` table hard-codes specific models (e.g. `MiniMax-M3`, `gpt-5.4-mini`). When you switch to a custom provider that **does not serve that model**, Codex's memory backend can't resolve it, so memory extraction / consolidation is skipped entirely.

### Fix: use the "Enable Codex memories" toggle

CC Switch v3.16.5+ adds a new **Enable Codex memories** checkbox in the Codex provider form's `config.toml (TOML)` editor row, placed as the **leftmost** of the four checkboxes (before **Enable Goal mode**).

**When checked**, cc-switch manages three things at once:

- `[memories].extract_model` and `[memories].consolidation_model`: written from the active provider's `model` (managed via "Model Mapping / Model Catalog"). After switching providers, the next save re-syncs to the new model.
- `[features].memories = true`: the Codex feature flag that actually turns the memory feature on. **This field is auto-written by the toggle** — no need to append it manually.
- All three are re-written whenever the active model changes.

**When unchecked**, cc-switch only removes those three keys:

- `[memories].extract_model` / `[memories].consolidation_model` are removed; other fields (`generate_memories` / `use_memories` / `disable_on_external_context` / `min_rate_limit_remaining_percent`) are preserved
- `[features].memories` is removed; other `[features]` keys (e.g. `web_search`) are preserved
- If `[memories]` becomes empty after the removal, the whole table is dropped (same for `[features]`)
- Codex falls back to the top-level `model` (default behavior)

> ⚠️ **Critical prerequisite**: Codex's memories / Chronicle feature is gated by a **feature flag**. CC Switch's toggle **auto-writes** `[features].memories = true`, so you do not need to add it manually. If you are using a manual `config.toml` setup outside of CC Switch, you still need:
>
> ```toml
> [features]
> memories = true
> ```
>
> Without it, Codex silently ignores the `[memories]` table at startup and the feature stays disabled.

### Steps

1. Open the Codex provider card you want to edit, click **Edit**
2. Make sure the **Model Mapping / Model Catalog** first row has a `model` filled in (cc-switch reads this value to sync `extract_model` / `consolidation_model`)
3. Check the **Enable Codex memories** checkbox
4. Click **Save**
5. **Restart Codex CLI** — Codex reads `config.toml` on startup; hot-reload won't pick up memory changes

### When to use it

| Scenario | Recommended |
|----------|-------------|
| OpenAI official provider | Optional (Codex already falls back to `model`; explicit is more stable) |
| Third-party gateway (DeepSeek / GLM / Qwen / self-hosted OpenAI-compatible) | **Recommended** to avoid model-name mismatches |
| Model Mapping has multiple distinct model names | **Recommended** so the memory model follows the chat model |
| You want a cheaper model for memory than for chat | **Don't check** — manually edit `config.toml` with fixed model names instead |

### Common misconceptions

- **"I can't set a custom extract_model after checking"** — by design: keeping the memory model aligned with the chat model avoids cross-provider failures. On every save, cc-syncs `[memories].extract_model` / `[memories].consolidation_model` from the first row of "Model Mapping / Model Catalog", **overwriting any value you edited manually in the TOML editor**. To mix models (e.g. cheaper memory than chat), uncheck the toggle and edit the two model fields in `config.toml` directly.
- **"Codex doesn't pick up the change"** — restart Codex CLI. It only reads `config.toml` at launch.
- **"Are Memories and Chronicle the same?"** — Chronicle is Codex's "cross-session narrative" feature and reuses the same `[memories].consolidation_model` field.
- **"Unchecking wiped my `generate_memories = true`"** — it doesn't. Unchecking only removes the two model fields in `[memories]` and the `memories` key in `[features]`; all other keys (e.g. `generate_memories`, `use_memories`, `web_search`) are preserved.
- **"I want a cheaper model for memory than chat"** — uncheck the toggle and edit the two model fields in `config.toml` directly. Checking the toggle will overwrite them with the top-level `model` on the next save.
