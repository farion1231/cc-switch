# 2.7 Codex 記憶 / Chronicle 機能

> 対応バージョン：CC Switch v3.16.5+（[OpenAI Codex 記憶ドキュメント](https://developers.openai.com/codex/memories) 参照）

Codex CLI には **Memories** と **Chronicle** 機能が組み込まれており、各セッション終了後に自動的に要点を抽出し、セッションを跨いで集約します。`~/.codex/config.toml` の `[memories]` 段で制御します：

```toml
[memories]
generate_memories = true                       # bool：記憶生成を有効化
use_memories = true                            # bool：新しいセッションに既存記憶を注入
disable_on_external_context = false            # bool：MCP / Web 検索など外部コンテキストを使用したセッションを記憶生成から除外
min_rate_limit_remaining_percent = 25          # int：記憶生成を開始する最低レート制限残量
extract_model = "MiniMax-M3"             # スレッド単位の記憶抽出モデル
consolidation_model = "gpt-5.4-mini"      # グローバル記憶の集約モデル
```

`extract_model` と `consolidation_model` を明示的に設定すると、Codex はそのモデルで記憶処理を行います。**未設定の場合はトップレベルの `model`（現在のプロバイダのメインモデル）にフォールバックします。**

## よくある問題：カスタムプロバイダで記憶機能がエラーになる

### 症状

サードパーティ / カスタムプロバイダに切り替え後、Codex の起動時に以下のようなエラーが出る：

```
Codex memory/chronicle backend failed: model "MiniMax-M3" not found
```

または `extract_model` / `consolidation_model` で「モデルが存在しない」と報告される。

### 原因

`[memories]` 段には特定のモデル名（例：`MiniMax-M3`、`gpt-5.4-mini`）がハードコードされています。**そのモデルを提供していない**カスタムプロバイダに切り替えると、Codex の記憶バックエンドがモデルを見つけられず、記憶の抽出/集約が完全にスキップされます。

### 解決：「Codex 記憶機能を有効化」トグルを使う

CC Switch v3.16.5+ では、Codex プロバイダ設定の **config.toml (TOML) エディタ**行に新しい「**Codex 記憶機能を有効化**」チェックボックスが追加されました。位置は 4 つのチェックボックスの中で**最も左**（「**Goal mode を有効化**」より前）です。

**チェックした場合**、cc-switch は 3 つを同時に管理します：

- `[memories].extract_model` と `[memories].consolidation_model`：現在のプロバイダの `model`（「モデルマッピング / モデルカタログ」で管理）を書き込みます。別プロバイダへの切り替え後、次回保存時に新モデルへ自動同期。
- `[features].memories = true`：Codex 記憶機能の feature flag。**このフィールドはトグルが自動書き込み**するため、手動追加は不要です。
- 上記 3 つは現在のモデルが変わるたびに再書き込みされます。

**チェックを外した場合**、cc-switch はこれら 3 つのキーのみを削除します：

- `[memories].extract_model` / `[memories].consolidation_model` が削除され、他のフィールド（`generate_memories` / `use_memories` / `disable_on_external_context` / `min_rate_limit_remaining_percent`）は保持されます
- `[features].memories` が削除され、段内の他のキー（例：`web_search`）は保持されます
- 削除後に `[memories]` 段が空になった場合、段ごと削除されます（`[features]` 段も同様）
- Codex はトップレベル `model` へのフォールバック（デフォルト動作）に戻ります

> ⚠️ **重要な前提条件**：Codex の記憶 / Chronicle 機能は **フィーチャーフラグ** で制御されています。CC Switch のトグルは `[features].memories = true` を**自動書き込み**するため、手動追加は不要です。CC Switch を使わず手動で `config.toml` を管理している場合は引き続き以下が必要です：
>
> ```toml
> [features]
> memories = true
> ```
>
> これがないと、Codex は起動時に `[memories]` 段を完全に無視し、記憶機能は無効のままです。

### 操作手順

1. 編集する Codex プロバイダのカードを開き、「編集」をクリック
2. 「モデルマッピング / モデルカタログ」の最初の行に `model` が入力されていることを確認（cc-switch が `extract_model` / `consolidation_model` を同期する際にこの値を読み取ります）
3. 「**Codex 記憶機能を有効化**」チェックボックスにチェックを入れる
4. 「保存」をクリック
5. **Codex CLI を再起動** — Codex は起動時に `config.toml` を読み取るため、ホットリロードでは反映されません

### 使い分け

| シナリオ | 推奨 |
|----------|------|
| OpenAI 公式プロバイダ | 任意（Codex はすでに `model` にフォールバック。明示した方が安定） |
| サードパーティゲートウェイ（DeepSeek / GLM / Qwen / 自前 OpenAI 互換） | **チェック推奨** — モデル名不一致を回避 |
| モデルマッピングに複数の異なるモデル名がある | **チェック推奨** — チャット用モデルに記憶モデルを追従 |
| チャットより安いモデルで記憶処理したい | **チェックしない** — `config.toml` を直接編集して固定モデル名を指定 |

### よくある質問

- **チェック後に独自の extract_model を設定できない？** — 仕様です：チャットと記憶のモデルを一致させることでプロバイダ間の不整合を回避しています。保存のたびに cc-switch は「モデルマッピング / モデルカタログ」の最初の行の `model` から `[memories].extract_model` / `[memories].consolidation_model` を同期するため、**TOML エディタで直接編集した値も上書きされます**。混合設定（チャットより安いモデルで記憶処理など）が必要な場合は、トグルのチェックを外し、`config.toml` の 2 つの model フィールドを直接編集してください。
- **変更が Codex に反映されない？** — Codex CLI を再起動してください。起動時のみ `config.toml` を読み込みます。
- **チェックを外したら自分で書いた `generate_memories = true` が消えた？** — 消えません。チェックを外すと `[memories].extract_model` / `[memories].consolidation_model` の 2 つと `[features].memories` のみが削除され、他のキー（例：`generate_memories` / `use_memories` / `web_search` など）は保持されます。
- **チャットより安いモデルで記憶処理したい** — トグルのチェックを外し、`config.toml` の 2 つの model フィールドを直接編集してください。トグルをオンにすると、次回保存時にトップレベル `model` で上書きされてしまいます。
- **Memories と Chronicle は別物？** — Chronicle は Codex の「セッション跨ぎナラティブ」機能で、同じ `[memories].consolidation_model` フィールドを使用します。
