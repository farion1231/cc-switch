## 一句话描述

**OmniAgent Workbench** 是一个 Fork 自 CC-Switch 的本地桌面应用，在 CC-Switch 的 Provider 管理 + 代理转发基础之上，新增 MiroFish 验证模式的编排引擎，让 Claude Code / Codex / OpenCode 等 AI 编程客户端的可验证任务优先经过**工具验证 + 多模型交叉检查 + 自动升级**，目标是在成本受控下达到或超过单次强模型；不可验证或高风险任务必须回退到强模型或人工闸门。

> 手动评审修正（2026-06-01）：本文档中的“超越大模型”只能按可验证工程任务、成本、延迟、回退率和人工验收率衡量，不能解释为在所有开放式任务上无条件超过最强前沿模型。

## 问题定义

### 现状痛点

开发者使用 Claude Code / Codex 等 AI 编程工具时：
1. **答案不可靠** — 单一模型偶尔产出幻觉代码，花更多时间审查和修复
2. **无法自动验证** — 生成的代码没有经过测试运行、lint 检查、schema 校验
3. **成本不透明** — 各自调用 API，没有统一的成本追踪和预算控制
4. **策略不可调** — 现有网关只做转发，无法定义「代码任务用 3 个模型辩论」这样的策略

### MiroFish 已验证的解决方案

MiroFish 的 ReACT 循环已在社交模拟预测中证明了核心模式：
- **最少 3 次工具调用** 强制执行 — 阻止 LLM 走捷径（v2 适配：异构验证步骤）
- **工具多样性跟踪** — 确保不重复使用同一个信息源
- **冲突解决** — 当 LLM 同时产出工具调用和最终答案时，优雅降级
- **温度衰减重试** — 每次重试降低温度 0.1，收敛到稳定输出
- **JSON 自我修复** — 多级解析策略，应对 LLM 格式不规整

```

        │
        └─ 编排模式开启 ↓
                │
                ▼
        Step 1: TaskClassifier — 分析任务类型
                ├─ 代码生成? → need_code = true
                ├─ 包含图片? → has_image = true
                ├─ 简单问题? → complexity = 0.2
                └─ 复杂架构? → complexity = 0.9
                │
                ▼
        Step 2: StrategySelector — 从 YAML 读取策略
                ├─ 读取 configs/strategies.yaml
                ├─ 匹配 TaskProfile → 选择策略
                └─ 策略可被 UI 或 YAML 修改
                │
                ▼
        Step 3: StrategyExecutor — 执行策略
                │
                ├─ ROUTE: 直接路由到最佳模型
                ├─ CASCADE: 便宜模型 → 质量检查 → 升级
                ├─ DEBATE: 多模型独立回答 → Judge 仲裁
                ├─ MoA: 多 Proposer → Aggregator
                └─ CROSS-MODAL: 视觉 + 文本分离执行
                │
                ▼
        Step 4: QualityGate — 工具验证 (MiroFish 模式)
                ├─ 代码? → 运行测试 + lint
                ├─ JSON? → Schema 校验
                ├─ 文本? → LLM 自评 + 交叉检查
                └─ 通过? → 返回 | 不通过? → 升级到更强模型
                │
                ▼
        Step 5: ResponseAdapter — 转换为客户端格式
                ├─ Claude Code → Anthropic SSE
                ├─ Codex → OpenAI SSE
                └─ 记录 CostLedger + JsonlLogger
```

## 低成本小模型组合超越单次大模型的能力路线

### 目标重新定义

本项目的长期目标不是简单地“把多个小模型相加变成一个大模型”，而是构建一个本地推理与验证平台：

> 使用低成本小模型集群，通过路由、候选生成、搜索、验证、排序、融合、回退和持续学习，在目标工作流上达到或超过单次调用强大模型的综合效果。

这里的“超过”必须用可观测指标定义，而不是主观体感：

- 质量：最终答案通过测试、编译、schema、人工评审或 benchmark 的比例更高。
- 成本：单位成功任务的平均 token/API 成本更低。
- 稳定性：失败时能自动发现、修复或回退，而不是把错误答案直接交给用户。
- 可解释性：系统能说明为什么选择某个模型、某个候选、某个策略。
- 覆盖率：在代码、配置、文档、工具调用、多步骤工程任务中稳定工作。

因此，更准确的目标是：

> 在可验证的工程任务上，用低成本模型组合和强验证系统实现大模型级能力；在不可验证或高风险任务上自动升级到强模型或人工闸门。

“全面替代最强大模型并在所有开放任务上稳定超越”不是第一阶段目标。小模型组合不能凭空获得所有小模型都没有的能力，必须依赖工具、搜索、验证、数据和回退机制补足。

### 理论基础

低成本模型组合路线理论上成立，前提是满足以下条件：

1. **错误不完全相关**：不同模型、不同 prompt、不同角色产生的错误不能高度一致，否则投票和融合没有意义。
2. **正确候选存在**：至少有一个候选答案接近正确，ranker 或 verifier 才可能选出来。
3. **验证器可靠**：系统必须能用测试、编译、schema、事实来源或强裁判判断候选质量。
4. **任务可分解**：复杂任务要能拆成规划、实现、审查、测试、修复、融合等步骤。
5. **预算可动态分配**：简单任务不能过度编排，复杂任务必须允许多轮搜索和升级。

相关研究方向已经证明了这条路线的可行性边界：

- Self-Consistency：多路径采样能提升推理任务表现。
- LLM-Blender：多模型候选排序和融合能优于单模型输出。
- RouteLLM：按任务路由模型可以降低成本并保持质量。
- Mixture-of-Agents：多层代理协作可以提升部分 benchmark 表现。
- Test-time compute scaling：在推理阶段增加搜索和验证，有时比单纯扩大模型参数更有效。

这些方向共同说明：提升能力的关键不是“模型数量”，而是“候选多样性 + 可靠排序 + 外部验证 + 动态预算”。

### 当前 v2 设计已覆盖的基础能力

当前设计已经包含通往该目标的第一层基础：

- Proxy 网关：统一截获 Claude Code、Codex、OpenCode 等客户端请求。
- TaskClassifier：识别任务类型、复杂度、风险和是否需要工具。
- StrategySelector：从 YAML 策略中选择 ROUTE、CASCADE、DEBATE、MoA 等策略。
- CASCADE：先用低成本模型，质量不足再升级。
- QualityGate：通过测试、lint、schema、编译和 LLM cross-check 进行质量门控。
- PredictiveRouter：基于历史学习预测模型成功率和成本。
- CrossJudge：用多 judge 交叉评分降低单一裁判偏差。
- HumanGate：对高风险任务引入人工确认。
- CostLedger：记录成本、延迟、模型选择和策略执行结果。

这些模块足以支撑 MVP 和第一阶段成本优化，但不足以支撑“全面超越强模型”的长期目标。

### 为实现更高目标仍需补齐的核心系统

#### 1. Benchmark 与评估系统

没有评估系统，就无法证明系统是否真的超过强模型。必须建立项目内 benchmark：

- 代码生成：函数实现、组件实现、接口封装。
- 代码修复：bug 定位、测试失败修复、类型错误修复。
- 重构迁移：跨文件改造、配置迁移、API 迁移。
- 架构设计：模块边界、数据流、风险识别。
- 文档任务：设计文档、用户手册、变更说明。
- 工具调用：命令执行、文件修改、测试验证。
- 长上下文任务：跨文件理解、历史对话恢复、项目级修改。

每个 benchmark 样本至少记录：

- 输入任务和约束。
- 参考答案或可验证验收条件。
- 单次强模型 baseline。
- 小模型组合输出。
- 成本、延迟、调用次数、失败次数。
- 是否通过测试或人工评审。
- 是否触发回退。

成功标准不能只看“答案看起来好”，必须看：

```text
quality_score
test_pass_rate
compile_pass_rate
human_accept_rate
cost_per_success
latency_p95
fallback_rate
regression_rate
```

#### 2. 强验证器系统

要超越强模型，不能只靠 LLM 评审 LLM。必须把真实验证作为核心能力：

- Rust/TypeScript/Python：编译、类型检查、单元测试、集成测试。
- JSON/YAML/TOML：schema 校验、解析校验、字段兼容校验。
- API：mock server、契约测试、请求/响应 golden test。
- 文档：引用检查、事实来源一致性、链接有效性。
- 安全：危险命令检测、权限边界检测、密钥泄漏扫描。
- UI：组件测试、可访问性检查、截图回归。

QualityGate 的长期形态应从“轻量文本质量分数”升级为：

```text
QualityGate =
  deterministic_checks
  + tool_verification
  + domain_specific_rules
  + ranker_score
  + judge_score
  + uncertainty_score
```

只有外部验证通过的候选，才允许进入最终融合或直接返回。

#### 3. 专用 Ranker/Judge

当前 QualityGate 只能做基础过滤，不能真正判断复杂答案优劣。长期需要一个专用排序器：

```text
input:
  task
  constraints
  candidate_answers[]
  tool_results[]
  historical_stats

output:
  ranked_candidates
  selected_candidate
  confidence
  risk_flags
  needs_fallback
```

Ranker 数据来源：

- 历史任务成败记录。
- 测试/编译/lint 结果。
- 人工接受或拒绝记录。
- 强模型裁判结果。
- 多 judge 交叉评分结果。

Ranker 可以先用规则 + LLM judge 混合实现，后续再训练 pairwise ranker 或 reward model。关键要求是：ranker 必须知道“哪些候选真的通过了验证”，而不是只看语言流畅度。

#### 4. 候选多样性生成系统

小模型组合要有效，必须避免多个模型给出高度相似的错误答案。候选生成需要主动制造多样性：

- 模型多样性：不同厂商、不同尺寸、不同训练风格。
- Prompt 多样性：直接实现、先写测试、先审查风险、先给方案、先找反例。
- 角色多样性：Implementer、Reviewer、Tester、Architect、Security Checker。
- 温度多样性：低温稳定候选 + 中温探索候选。
- 路径多样性：一次性回答、分步计划、工具验证、反例驱动修复。

候选生成不应只是：

```text
same_prompt -> model_a
same_prompt -> model_b
same_prompt -> model_c
```

而应是：

```text
task -> candidate_plan[]
task + plan_i -> implementation_i
implementation_i -> test_i
implementation_i + test_result_i -> repair_i
all_verified_candidates -> ranker -> fusion
```

#### 5. Test-time Search 推理搜索

“超越单次强模型”的关键在于测试时搜索，而不是一次性调用更多模型。

长期流程应升级为：

```text
Generate candidates
-> Verify candidates
-> Rank candidates
-> Repair weak candidates
-> Re-verify
-> Fuse best candidates
-> Final validate
-> Return or fallback
```

搜索策略包括：

- Best-of-N：生成 N 个候选，选择最优。
- Self-Consistency：多条推理路径投票。
- Tree/Graph of Thoughts：保留中间状态，扩展高价值分支。
- Debate：正反方指出缺陷。
- Reflexion：根据失败验证结果自我修复。
- Cascade Repair：便宜模型失败后，让中等模型修复，而不是从零重写。

该系统必须受预算控制，不允许无限搜索。

#### 6. 动态预算规划器

当前 YAML 策略是静态规则。要达到更高能力，需要 BudgetPlanner：

```rust
pub struct BudgetPlan {
    pub max_cost_usd: f64,
    pub max_latency_ms: u64,
    pub candidate_count: u32,
    pub max_repair_rounds: u32,
    pub allow_frontier_fallback: bool,
    pub require_human_gate: bool,
    pub verification_level: VerificationLevel,
}
```

预算应由以下因素决定：

- 任务复杂度。
- 风险等级。
- 用户是否要求低延迟。
- 是否可验证。
- 历史上类似任务的小模型成功率。
- 当前 provider 健康状态。
- 当前成本预算余额。

低风险简单任务：单小模型直接回答。  
中风险工程任务：小模型候选 + 工具验证 + ranker。  
高风险任务：多候选 + 强验证 + 强模型抽检 + 人工闸门。  
不可验证任务：限制自动化承诺，提高 fallback 概率。

#### 7. 持续学习闭环

系统必须从每次请求中学习，否则策略永远停留在手写规则。

每次执行都应写入 telemetry：

```json
{
  "task_type": "coding",
  "complexity": 0.72,
  "risk": "medium",
  "strategy": "cascade",
  "models_used": ["cheap_coder", "mid_coder"],
  "candidate_count": 3,
  "verification_results": {
    "cargo_test": "pass",
    "eslint": "pass",
    "schema": "not_applicable"
  },
  "ranker_score": 0.84,
  "judge_score": 0.79,
  "human_accepted": true,
  "fallback_used": false,
  "cost_usd": 0.012,
  "latency_ms": 18400
}
```

这些数据用于训练或调优：

- Router：类似任务该走哪个策略。
- Ranker：哪个候选更可能通过验收。
- ThresholdOptimizer：不同任务类型的质量阈值。
- BudgetPlanner：复杂任务该分配多少候选和验证轮次。
- ModelHealth：某模型近期在哪类任务上退化。

没有 telemetry，系统无法系统性提升。

### 升级后的长期架构

长期目标架构应从当前：

```text
Request
-> TaskClassifier
-> StrategySelector
-> StrategyExecutor
-> QualityGate
-> ResponseAdapter
```

升级为：

```text
Request
-> TaskClassifier
-> RiskAnalyzer
-> BudgetPlanner
-> StrategySelector
-> CandidateGenerator
-> ToolVerifier
-> Ranker
-> FusionEngine
-> FinalValidator
-> FallbackController / HumanGate
-> ResponseAdapter
-> TelemetryLearning
```

各模块职责：

- `TaskClassifier`：识别任务类型、复杂度、上下文形态。
- `RiskAnalyzer`：判断是否涉及文件删除、配置破坏、安全风险、生产影响。
- `BudgetPlanner`：决定成本、延迟、候选数、验证强度和是否允许强模型。
- `CandidateGenerator`：用多个小模型和多种 prompt 生成候选。
- `ToolVerifier`：运行真实测试、编译、schema、lint、mock API。
- `Ranker`：基于任务、候选和验证结果排序。
- `FusionEngine`：融合多个高质量候选，而不是盲目总结所有候选。
- `FinalValidator`：最终门禁，决定是否返回、修复、升级或请求人工确认。
- `FallbackController`：低置信度时升级到强模型或人工闸门。
- `TelemetryLearning`：记录结果并反哺路由、预算、阈值和 ranker。

### 根本缺陷与不可消除风险

即使完成上述升级，系统仍存在根本限制：

1. **能力上限限制**：如果所有小模型都无法产生正确思路，组合系统通常无法凭空创造正确答案。
2. **裁判瓶颈**：ranker/judge 质量决定最终选择质量，弱裁判会偏爱流畅但错误的答案。
3. **错误相关性**：同源训练数据和相似模型架构会导致相同错误，投票会放大错误共识。
4. **验证覆盖不足**：很多开放式任务无法完全自动验证，系统只能估计质量。
5. **成本与延迟膨胀**：多候选、多轮修复和多 judge 会显著增加响应时间。
6. **复杂度上升**：编排系统本身会成为需要测试、监控和调试的大型软件。
7. **上下文污染**：多个候选互相引用或融合时，错误信息可能被带入最终答案。
8. **安全风险**：自动执行工具验证时必须严格限制命令、路径、权限和敏感信息。
9. **评估漂移**：benchmark 可能被系统过拟合，导致真实任务效果不如评估结果。

因此，产品承诺必须保持严谨：

- 可以承诺“更高性价比”和“可验证任务上的质量提升”。
- 可以承诺“失败可检测、可回退、可审计”。
- 不应承诺“所有任务全面超过最强模型”。

### 分阶段增强路线

#### Phase A: MVP 可靠闭环

- ROUTE/CASCADE。
- 确定性 QualityGate。
- 非流式 CASCADE 接管。
- Runtime YAML 开关。
- 基础单元测试和组件测试。

目标：证明本地网关能安全接管部分请求，失败时可靠 fallback。

#### Phase B: 可验证任务增强

- 工具验证器接入。
- 编译、测试、schema、lint 结果入库。
- Best-of-N 小模型候选。
- 简单 ranker。

目标：在代码和配置任务上稳定超过单次小模型，并接近强模型。

#### Phase C: Ranker 与 Benchmark

- 项目内 benchmark。
- 强模型 baseline 对比。
- 历史 telemetry。
- Pairwise ranking 数据集。
- 任务类型级质量阈值。

目标：可以量化回答“在哪些任务上超过强模型，成本降低多少”。

#### Phase D: Test-time Search

- 多轮修复。
- Reflexion。
- Debate/MoA。
- FusionEngine。
- 动态预算。

目标：在复杂工程任务上超过单次强模型输出。

#### Phase E: 自学习与自优化

- PredictiveRouter 训练。
- BudgetPlanner 学习。
- ThresholdOptimizer。
- ModelHealth 自动降权。
- 人工反馈闭环。

目标：系统随使用持续提升，而不是依赖固定 YAML 规则。

### 具体实现路径与工程方法

实现“低成本小模型组合达到或超过单次强模型”的路径必须分成可验证的工程闭环。每一阶段都要能独立上线、独立评估、独立回退。

#### Step 0: 建立强模型 Baseline

在优化小模型组合前，必须先定义强模型基线，否则无法判断是否真的超越。

实现方法：

1. 选定 frontier baseline，例如 Claude Opus/Sonnet、GPT-5 级别模型或用户当前最信任的单模型。
2. 建立固定 benchmark 任务集，至少覆盖代码、配置、文档、架构、工具调用五类任务。
3. 对每个任务保存：
   - 原始 prompt。
   - 仓库上下文或测试 fixture。
   - 强模型输出。
   - 成本、延迟、token 使用。
   - 测试/编译/人工评审结果。
4. 后续所有小模型组合策略都必须和该 baseline 比较。

落地模块：

```text
src-tauri/src/eval/
  benchmark.rs
  baseline_runner.rs
  task_fixture.rs
  scoring.rs
```

数据表：

```sql
benchmark_tasks(id, category, prompt, fixture_path, expected_checks, difficulty, risk)
benchmark_runs(id, task_id, system_name, strategy, models_used, cost_usd, latency_ms, score, passed, created_at)
```

#### Step 1: 先做可验证任务，而不是开放式任务

第一批目标必须选择“有客观验证信号”的任务，因为小模型组合最依赖 verifier。

优先级：

1. 代码修复：是否通过测试。
2. 配置生成：是否通过 schema 和解析。
3. 类型/编译错误修复：是否通过 `cargo check`、`tsc`。
4. API 适配：是否通过 mock contract test。
5. 文档生成：是否通过链接、引用、结构检查，再加人工评分。

暂缓：

- 开放式创意写作。
- 无参考答案的战略判断。
- 纯审美类 UI 评价。
- 需要高事实准确性但没有来源校验的数据问答。

工程原则：

```text
No verifier -> no claim of superiority.
```

没有验证器的任务可以执行，但不能计入“超过强模型”的核心指标。

#### Step 2: 实现 CandidateGenerator

CandidateGenerator 的目标不是简单调用多个模型，而是制造互补候选。

候选类型：

```rust
pub enum CandidateRole {
    DirectAnswer,
    PlanThenAnswer,
    TestFirst,
    Reviewer,
    EdgeCaseFinder,
    SecurityChecker,
    RefactorSpecialist,
}
```

实现方法：

1. 根据 `TaskProfile` 和 `BudgetPlan` 选择候选数量。
2. 为每个候选分配不同 role、prompt template、temperature、model。
3. 每个候选输出统一结构：

```json
{
  "candidate_id": "c_001",
  "role": "test_first",
  "model": "cheap_coder",
  "content": "...",
  "claimed_changes": ["src/foo.rs"],
  "confidence": 0.62,
  "assumptions": ["..."],
  "known_risks": ["..."]
}
```

4. 候选不能直接返回用户，必须进入 verifier/ranker。

落地模块：

```text
src-tauri/src/orchestration/candidate_generator.rs
src-tauri/src/orchestration/prompt_templates.rs
src-tauri/src/orchestration/candidate.rs
```

#### Step 3: 实现 ToolVerifier

ToolVerifier 是能力提升的核心，不是附属功能。

验证级别：

```rust
pub enum VerificationLevel {
    None,
    DeterministicOnly,
    StaticChecks,
    FullLocalChecks,
    FullWithHumanGate,
}
```

验证器接口：

```rust
pub trait Verifier: Send + Sync {
    fn name(&self) -> &'static str;
    fn can_verify(&self, task: &TaskProfile, candidate: &Candidate) -> bool;
    async fn verify(&self, task: &TaskProfile, candidate: &Candidate) -> VerificationResult;
}
```

核心验证器：

- `SchemaVerifier`: JSON/YAML/TOML 解析和 schema。
- `CompileVerifier`: `cargo check`、`tsc --noEmit`。
- `TestVerifier`: 指定测试或 affected tests。
- `LintVerifier`: clippy、eslint、ruff。
- `PatchVerifier`: diff 是否只改允许文件。
- `SafetyVerifier`: 删除、密钥、危险命令、路径越界。

输出结构：

```json
{
  "verifier": "cargo_test",
  "status": "pass",
  "score": 1.0,
  "duration_ms": 18400,
  "evidence": "cargo test orchestration --lib passed",
  "failure_summary": null
}
```

注意：

- 本地命令验证必须有权限边界。
- 不允许 verifier 自动执行破坏性命令。
- 所有 verifier 输出都必须进入 ranker 和 telemetry。

#### Step 4: 实现 Ranker v1

Ranker v1 不需要一开始训练模型，先用规则 + LLM judge 混合。

排序公式：

```text
candidate_score =
  0.45 * verifier_score
  + 0.20 * task_fit_score
  + 0.15 * risk_penalty_inverse
  + 0.10 * model_historical_success
  + 0.10 * judge_score
```

硬性规则：

- 有严重安全风险的候选直接淘汰。
- 编译/测试失败的代码候选不能直接返回，只能进入 repair。
- verifier 通过的候选优先级高于语言流畅的候选。
- judge_score 不能覆盖真实测试失败。

落地模块：

```text
src-tauri/src/orchestration/ranker.rs
src-tauri/src/orchestration/scoring.rs
```

#### Step 5: 实现 Repair Loop

小模型组合超越单次强模型的关键之一，是能利用验证失败信息自动修复。

流程：

```text
candidate
-> verifier fails
-> failure summary
-> repair prompt
-> patched candidate
-> re-verify
-> rank again
```

修复策略：

- 低风险失败：同一小模型修复 1 次。
- 中风险失败：升级到 mid model 修复。
- 高风险失败：frontier fallback 或 HumanGate。
- 同一错误重复 2 次：停止修复，避免循环。

状态结构：

```rust
pub struct RepairState {
    pub candidate_id: String,
    pub round: u32,
    pub previous_failures: Vec<VerificationResult>,
    pub repaired_by_model: String,
    pub stopped_reason: Option<String>,
}
```

#### Step 6: 实现 FusionEngine

Fusion 不是把所有候选平均总结，而是只融合经过验证和排序的高质量候选。

输入：

- Top 2-3 个候选。
- 每个候选的 verifier evidence。
- 每个候选的已知风险。
- 用户原始约束。

融合规则：

- 不采纳 verifier 失败的代码片段。
- 保留通过测试的实现主体。
- 只吸收 reviewer/edge-case 候选中被 verifier 或 ranker 支持的建议。
- 输出必须再次经过 FinalValidator。

Fusion prompt 必须包含：

```text
Use only verified facts and verified code paths.
Do not include a change unless it is supported by verification evidence.
If candidates conflict, prefer the candidate with stronger verifier evidence.
```

#### Step 7: 实现 FinalValidator 与 FallbackController

FinalValidator 决定是否可以返回用户。

返回条件：

```text
verifier_score >= threshold
AND risk_score <= allowed_risk
AND uncertainty_score <= threshold
AND no blocking safety flag
```

Fallback 触发条件：

- 无候选通过最低验证门槛。
- ranker 最高分低于阈值。
- judge 之间分歧过大。
- repair loop 达到上限。
- 高风险任务缺少强验证证据。
- 用户配置要求高可靠模式。

Fallback 动作：

1. 升级更强模型。
2. 请求人工确认。
3. 返回“不确定，建议人工审查”的保守结果。
4. 回到原始 proxy passthrough。

#### Step 8: 实现 TelemetryLearning

所有执行路径都要记录，否则无法优化。

最小事件流：

```text
request_received
classification_done
budget_planned
candidate_generated
verification_done
ranking_done
repair_done
fusion_done
final_validation_done
response_returned
user_feedback_received
```

每条事件必须包含：

- `request_id`
- `task_type`
- `strategy`
- `model`
- `cost`
- `latency`
- `score`
- `pass/fail`
- `error`
- `evidence`

学习闭环：

```text
telemetry
-> aggregate stats
-> model success table
-> route policy update
-> budget threshold update
-> ranker feature update
```

初期不需要自动训练，可以先做统计驱动：

- 某模型在某任务类型连续失败，自动降权。
- 某策略成本高但质量无提升，降低触发概率。
- 某验证器误报率高，降低权重。
- 某类任务 fallback 率高，提高默认预算。

### 评估指标体系

评估必须同时覆盖质量、成本、延迟、稳定性、安全和学习效果。单一质量分数不能代表系统成功。

#### 一级指标：是否真的超过强模型

| 指标 | 定义 | 目标 |
|---|---|---|
| `quality_win_rate` | 小模型组合输出优于强模型 baseline 的比例 | Phase B >= 30%，Phase D >= 55% |
| `quality_tie_or_win_rate` | 小模型组合持平或优于强模型 baseline 的比例 | Phase B >= 70%，Phase D >= 85% |
| `cost_per_success_ratio` | 小模型组合每个成功任务成本 / 强模型每个成功任务成本 | <= 0.4 |
| `verified_success_rate` | 最终输出通过 verifier 的比例 | 可验证任务 >= 85% |
| `frontier_fallback_rate` | 需要强模型兜底的比例 | Phase B <= 35%，Phase D <= 20% |

#### 二级指标：工程可靠性

| 指标 | 定义 | 目标 |
|---|---|---|
| `compile_pass_rate` | 代码任务最终编译通过率 | >= 90% |
| `test_pass_rate` | 有测试任务最终测试通过率 | >= 85% |
| `schema_pass_rate` | 结构化输出 schema 通过率 | >= 98% |
| `unsafe_action_block_rate` | 危险动作被拦截比例 | 100% |
| `regression_rate` | 新策略低于旧策略的比例 | <= 5% |

#### 三级指标：成本与延迟

| 指标 | 定义 | 目标 |
|---|---|---|
| `avg_cost_usd` | 平均请求成本 | 按任务级预算配置 |
| `p50_latency_ms` | 中位响应时间 | 简单任务接近原 passthrough |
| `p95_latency_ms` | 95 分位响应时间 | 不超过用户预算 |
| `tokens_per_success` | 每个成功任务消耗 token | 持续下降 |
| `wasted_candidate_rate` | 生成但完全无贡献候选比例 | <= 30% |

#### 四级指标：Ranker 与 Judge 质量

| 指标 | 定义 | 目标 |
|---|---|---|
| `ranker_top1_accuracy` | ranker 选中的候选是否为最终最佳候选 | >= 70% |
| `ranker_top3_recall` | 最佳候选是否在 ranker top3 | >= 90% |
| `judge_disagreement_rate` | 多 judge 分歧超过阈值的比例 | 可观测并触发 fallback |
| `false_pass_rate` | verifier/judge 通过但人工或测试后失败 | <= 5% |
| `false_block_rate` | 实际可接受却被系统阻断 | <= 10% |

#### 五级指标：学习效果

| 指标 | 定义 | 目标 |
|---|---|---|
| `routing_improvement` | 学习后路由成功率提升 | 持续正向 |
| `threshold_stability` | 阈值调整后质量波动 | 不产生明显回退 |
| `model_health_precision` | 被降权模型是否确实表现退化 | >= 80% |
| `fallback_precision` | fallback 是否真的必要 | >= 70% |
| `feedback_acceptance_rate` | 用户接受系统最终输出比例 | 持续提升 |

### 评分计算方法

每次任务运行得到一个统一 `RunScore`：

```text
RunScore =
  0.40 * quality_score
  + 0.20 * verifier_score
  + 0.15 * cost_score
  + 0.10 * latency_score
  + 0.10 * safety_score
  + 0.05 * user_feedback_score
```

其中：

```text
cost_score = min(frontier_baseline_cost / actual_cost, 1.0)
latency_score = min(latency_budget_ms / actual_latency_ms, 1.0)
safety_score = 0 if any blocking safety violation else 1
verifier_score = weighted average of tool verifier results
quality_score = benchmark score or judge/ranker score
```

任务级胜负判断：

```text
small_model_system wins if:
  RunScore(system) > RunScore(frontier_baseline) + margin

tie if:
  abs(RunScore(system) - RunScore(frontier_baseline)) <= margin

loss if:
  RunScore(system) < RunScore(frontier_baseline) - margin
```

默认：

```text
margin = 0.03
```

对于代码任务，测试/编译失败时：

```text
verifier_score = 0
quality_score cannot exceed 0.4
```

这保证语言流畅度不能掩盖工程失败。

### 阶段验收门槛

#### Phase A 验收

- ROUTE/CASCADE 非流式链路可用。
- streaming 请求 100% passthrough。
- CASCADE 失败时 100% fallback，不破坏原 proxy。
- 基础 QualityGate 单元测试通过。
- UI 开关能读写 runtime YAML。

#### Phase B 验收

- 至少 50 个可验证 benchmark 样本。
- 可验证任务 `verified_success_rate >= 75%`。
- `cost_per_success_ratio <= 0.6`。
- 小模型组合相对单小模型 `quality_win_rate >= 40%`。
- 强模型 fallback 可观测并可配置。

#### Phase C 验收

- 至少 200 个 benchmark 样本。
- 有强模型 baseline。
- `ranker_top3_recall >= 85%`。
- telemetry 能生成模型/任务成功率表。
- 能回答“哪类任务小模型组合更划算”。

#### Phase D 验收

- 至少 500 个 benchmark 样本。
- 可验证工程任务 `quality_tie_or_win_rate >= 85%`。
- 对强模型 baseline `quality_win_rate >= 55%`。
- `cost_per_success_ratio <= 0.4`。
- `false_pass_rate <= 5%`。

#### Phase E 验收

- 路由策略经过 telemetry 自动调优。
- 模型健康降权能减少失败率。
- 阈值调整不引入质量回退。
- 用户接受率连续提升。
- 系统能自动生成策略改进建议。

### 成功判定

长期目标是否达成，必须用数据判断：

```text
For target task set:
  small_model_system_quality >= frontier_single_model_quality
  AND cost_per_success <= 40% of frontier_single_model_cost
  AND fallback_rate is controlled
  AND p95_latency is acceptable
  AND regression_rate does not increase
```

在可验证工程任务上，如果系统能以显著更低成本达到或超过强模型 baseline，并且失败时可检测、可回退、可审计，则认为“低成本小模型组合实现大模型级能力”的目标达成。

## MiroFish 核心模式移植

### 1. ReACT 循环 (from report_agent.py)

**MiroFish 原始实现：**
- 最少 3 次工具调用强制执行（lines 1377-1389）
- 工具多样性跟踪，提示未使用的工具（lines 1448-1454）
- 冲突解决：同时出现工具调用和 Final Answer 时，先重试 2 次，第 3 次降级（lines 1329-1361）
- 最大 5 轮迭代，超出后强制终止
- 多格式解析：XML 标签 > 裸 JSON 兜底（lines 1067-1125）

**OmniAgent Rust 移植：**

```rust
// orchestration/react_executor.rs

pub struct ReACTExecutor {
    max_iterations: u32,            // 默认 5
    min_verification_steps: u32,    // 默认 3 — 至少 3 个异构验证步骤
    max_conflict_retries: u32,      // 默认 2
    all_tools: HashSet<String>,
}

pub struct ReACTResult {
    pub answer: String,
    pub tool_calls_made: u32,
    pub tools_used: HashSet<String>,
    pub iterations: u32,
    pub quality_passed: bool,
}

impl ReACTExecutor {
    pub async fn execute(&self, task: TaskProfile) -> Result<ReACTResult> {
        let mut tool_calls_count = 0;
        let mut used_tools = HashSet::new();
        let mut conflict_retries = 0;

        for iteration in 0..self.max_iterations {
            let response = self.call_model(task.clone()).await?;

            // MiroFish 模式: 检测冲突 (tool_call + Final Answer 同时出现)
            if response.has_tool_call() && response.has_final_answer() {
                conflict_retries += 1;
                if conflict_retries >= self.max_conflict_retries {
                    // 第 3 次冲突: 强制降级，只取工具调用
                    return self.force_tool_only(response);
                }
                continue; // 重试
            }

            // 最少验证步骤强制（MiroFish 模式适配：v2 为异构验证步骤，非外部工具调用）
            if response.has_final_answer() && tool_calls_count < self.min_verification_steps {
                let unused = self.all_tools.difference(&used_tools);
                // 提示使用未使用的验证工具
                self.inject_tool_hint(unused);
                continue;
            }

            if response.has_tool_call() {
                tool_calls_count += 1;
                used_tools.insert(response.tool_name().clone());
                // MiroFish 模式: 跟踪工具多样性
                self.execute_tool(response.tool_call()).await?;
            }

            if response.has_final_answer() {
                return Ok(ReACTResult {
                    answer: response.final_answer(),
                    tool_calls_made: tool_calls_count,
                    tools_used,
                    iterations: iteration + 1,
                    quality_passed: false, // 后续由 QualityGate 判定
                });
            }
        }

        // MiroFish 模式: 强制终止
```

```rust
pub enum VerificationTool {

impl QualityGate {
    pub async fn verify(&self, response: &str, task: &TaskProfile) -> QualityResult {
        let mut scores = Vec::new();

        for tool in &self.tools {
            let score = match tool {
                VerificationTool::TestRunner => self.run_tests(response).await?,
                VerificationTool::Linter => self.run_linter(response).await?,
                VerificationTool::SchemaValidator => self.validate_schema(response).await?,
                VerificationTool::CompileChecker => self.compile_check(response).await?,
                VerificationTool::LLMCrossCheck => self.llm_cross_check(response).await?,
            };
            scores.push(score);
        }

        let avg = scores.iter().sum::<f64>() / scores.len() as f64;
        QualityResult {
            passed: avg >= self.pass_threshold,
            score: avg,
            individual_scores: scores,
        }
    }
```

```rust
pub fn heal_json(raw: &str) -> Result<Value> {
    // Level 1: 直接解析
    if let Ok(v) = serde_json::from_str(raw) {
        return Ok(v);
    }

    // Level 2: 关闭未闭合的括号 (from MiroFish _fix_truncated_json)
    let healed = close_unclosed_brackets(raw);
    if let Ok(v) = serde_json::from_str(&healed) {
        return Ok(v);
    }

    // Level 3: 提取 JSON 块 (from MiroFish _try_fix_config_json)
    let extracted = extract_json_block(raw);
    if let Ok(v) = serde_json::from_str(&extracted) {
        return Ok(v);
    }

    // Level 4: 移除控制字符 + 正则化空白
    let cleaned = clean_control_chars(&extracted);
    serde_json::from_str(&cleaned)
        .map_err(|e| anyhow!("JSON healing failed after 4 levels: {}", e))
}
```

```rust
// orchestration/strategy_loader.rs

pub struct StrategyLoader {
    path: PathBuf,
    last_modified: SystemTime,
    strategies: Arc<RwLock<StrategyConfig>>,
}

impl StrategyLoader {
    pub async fn watch(&self) {
        let mut interval = tokio::time::interval(Duration::from_secs(5));
        loop {
            interval.tick().await;
            let modified = fs::metadata(&self.path)
                .ok()
                .and_then(|m| m.modified().ok());

            if modified != Some(self.last_modified) {
                match self.reload() {
                    Ok(new_config) => {
                        *self.strategies.write().await = new_config;
                        self.last_modified = modified.unwrap();
                        // 通知 UI 策略已更新
                        self.notify_ui_reload();
                    }
                    Err(e) => {
                        // 加载失败，保持旧配置，通知 UI 错误
                        self.notify_ui_error(e);
                    }
                }
            }
        }
    }
}
```

### 预测能力（Predictive Routing）

```rust
// orchestration/predictive_router.rs

/// 基于历史数据预测本次请求的最优策略
pub struct PredictiveRouter {
    stats: Arc<StatsEngine>,
    db: Arc<SqlitePool>,
}

impl PredictiveRouter {
    /// 在 TaskClassifier 输出后，用历史数据增强决策
    ///
    /// 输入：TaskProfile（来自 TaskClassifier）
    /// 输出：PredictedBestStrategy（包含历史支撑数据）
    pub async fn predict(&self, profile: &TaskProfile) -> PredictedBestStrategy {
        // 1. 查找历史相似请求（相同 task_type + 相近 complexity）
        let similar = self.find_similar_requests(profile).await;

        // 2. 冷启动防护：历史数据不足时直接回退到规则匹配
        const MIN_SAMPLES: usize = 30;
        if similar.len() < MIN_SAMPLES {
            return PredictedBestStrategy {
                strategy: "yaml_default".to_string(),
                confidence: 0.0,
                sample_size: similar.len(),
                expected_quality: 0.0,
                expected_cost: 0.0,
                reasoning: format!(
                    "历史数据不足（{} < {}），回退到 StrategySelector 规则匹配",
                    similar.len(), MIN_SAMPLES
                ),
            };
        }

        // 3. 统计各策略在相似请求上的表现
        let strategy_stats = self.aggregate_strategy_performance(&similar);

        // 4. 选择历史表现最好的策略
        let best = strategy_stats.best_strategy();

        PredictedBestStrategy {
            strategy: best.name,
            confidence: best.confidence,
            sample_size: best.sample_count,
            expected_quality: best.avg_quality,
            expected_cost: best.avg_cost,
            reasoning: format!(
                "基于 {} 条相似请求，{} 策略平均质量 {}，成本 ${:.4}",
                best.sample_count, best.name, best.avg_quality, best.avg_cost
            ),
        }
```

## 随机编排与防偏差机制

### 设计目标

LLM-as-Judge 存在已知的**位置偏差**（Position Bias）——Judge 模型倾向于选择第一个或最后一个候选答案。MiroFish 设计文档（`MOA多模型方案B.txt` 第 239 行）明确要求"随机打乱候选顺序"。

OmniAgent 在 DEBATE 和 MoA 策略中引入**多层防偏差机制**，确保 Judge 仲裁的公平性和可靠性。

### 架构

```
┌─────────────────────────────────────────────────────────┐
│                 BiasMitigation                           │
│                                                          │
│  ┌───────────────┐  ┌───────────────┐  ┌─────────────┐ │
│  │ 候选随机打乱   │  │ 多 Judge 交叉 │  │ 强模型抽检  │ │
│  │ Shuffle        │  │ Cross-Judge   │  │ Spot-Check  │ │
│  └───────┬───────┘  └───────┬───────┘  └──────┬──────┘ │
│          │                  │                  │        │
│          └──────────────────┼──────────────────┘        │
│                             ▼                           │
│                    ┌────────────────┐                   │
│                    │ 共识检测器      │                   │
│                    │ Consensus      │                   │
│                    │ Detector       │                   │
```

### 候选随机打乱

```rust
// orchestration/shuffle.rs

use rand::seq::SliceRandom;

/// DEBATE/MoA 策略中，候选答案在交给 Judge 之前必须随机打乱
///
/// 理论依据：LLM Judge 存在位置偏差（Position Bias）
/// 来源：MiroFish MOA多模型方案B.txt line 239
pub struct CandidateShuffler;

impl CandidateShuffler {
    /// 打乱候选答案的顺序，并记录原始位置映射
    pub fn shuffle(candidates: Vec<CandidateAnswer>) -> ShuffledCandidates {
        let mut indexed: Vec<(usize, CandidateAnswer)> = candidates
            .into_iter()
            .enumerate()
            .collect();

        let mut rng = rand::thread_rng();
        indexed.shuffle(&mut rng);

        let order_map: HashMap<usize, usize> = indexed
            .iter()
            .enumerate()
            .map(|(new_pos, (orig_pos, _))| (*orig_pos, new_pos))
            .collect();

        ShuffledCandidates {
            candidates: indexed.into_iter().map(|(_, c)| c).collect(),
            order_map,
        }
    }
}

pub struct ShuffledCandidates {
    pub candidates: Vec<CandidateAnswer>,
    pub order_map: HashMap<usize, usize>,  // orig_pos → new_pos
}

#[derive(Debug, Clone)]
pub struct CandidateAnswer {
    pub model_key: String,
    pub content: String,
    pub quality_score: f64,
    pub latency_ms: u64,
    pub cost_usd: f64,
}
```

### 强模型抽检

```rust
// orchestration/spot_check.rs

/// 用强模型对 Judge 结果进行随机抽检
///
/// 逻辑：
/// 1. 每次编排有 10% 概率触发抽检（可配置）
/// 2. 抽检时，将原始问题 + 候选答案 + Judge 结果发给强模型
/// 3. 强模型判断 Judge 的选择是否正确
/// 4. 如果强模型不同意 → 记录到偏差日志 + 调整该 Judge 的权重
pub struct SpotChecker {
    inspector_model: String,       // 强模型，如 "frontier"
    check_probability: f64,        // 默认 0.1（10% 抽检率）
    db: Arc<SqlitePool>,
}

pub struct SpotCheckResult {
    pub triggered: bool,
    pub judge_correct: Option<bool>,   // Some(true/false) 如果触发了抽检
    pub inspector_reasoning: Option<String>,
}

impl SpotChecker {
    pub async fn maybe_check(
        &self,
        prompt: &str,
        candidates: &[CandidateAnswer],
        judge_result: &CrossJudgeResult,
    ) -> Result<SpotCheckResult> {
        let mut rng = rand::thread_rng();
        if rng.gen::<f64>() > self.check_probability {
            return Ok(SpotCheckResult {
                triggered: false,
                judge_correct: None,
                inspector_reasoning: None,
            });
        }

        // 调用强模型评估
        let inspection = self.call_inspector(prompt, candidates, judge_result).await?;

        // 记录到偏差日志
        if !inspection.judge_agrees {
            self.record_bias(prompt, judge_result, &inspection).await?;
        }

        Ok(SpotCheckResult {
            triggered: true,
            judge_correct: Some(inspection.judge_agrees),
            inspector_reasoning: Some(inspection.reasoning),
        })
    }
}
```

## 动态工作流部署

### 设计目标

OmniAgent 的工作流不是静态 YAML 定义的固定流程，而是根据**任务特征、模型表现、预算状态**动态组装的。每次请求的工作流可能不同。

### 成本-质量评分函数

来自 MiroFish 设计文档（`MOA多模型方案B.txt` 第 244-250 行）的核心公式：

```
Score(model, task) = ExpectedQuality - λ × Cost - μ × Latency - ρ × Risk + τ × ToolVerifiability
```

```rust
// orchestration/scoring.rs

/// 成本-质量综合评分
pub struct CostQualityScorer {
    /// 质量权重（默认 0.5）
    quality_weight: f64,
    /// 成本敏感系数（默认 0.3）
    cost_sensitivity: f64,
    /// 延迟敏感系数（默认 0.1）
    latency_sensitivity: f64,
    /// 风险惩罚系数（默认 0.2）
    risk_penalty: f64,
    /// 可验证性奖励系数（默认 0.15）
    verifiability_bonus: f64,
}

pub struct ModelScore {
    pub model_key: String,
    pub score: f64,
    pub breakdown: ScoreBreakdown,
}

pub struct ScoreBreakdown {
    pub expected_quality: f64,    // 基于历史数据
    pub cost_term: f64,           // 归一化成本
    pub latency_term: f64,        // 归一化延迟
    pub risk_term: f64,           // 风险惩罚
    pub verifiability_term: f64,  // 可验证性奖励
}

impl CostQualityScorer {
    /// 计算模型在给定任务上的综合评分
    pub fn score(
        &self,
        model_key: &str,
        profile: &TaskProfile,
        history: &ModelStats,
        budget_remaining: f64,
    ) -> ModelScore {
        let quality = history.avg_quality;

        // 成本归一化：相对日预算的比例
        let cost_normalized = (history.avg_cost_usd / budget_remaining).min(1.0);

        // 延迟归一化：相对 30s 上限
        let latency_normalized = (history.avg_latency_ms as f64) / 30000.0;

        // 风险惩罚：critical 任务额外扣分
        let risk = match profile.risk {
            RiskLevel::Critical => 1.0,
            RiskLevel::High => 0.6,
            RiskLevel::Medium => 0.3,
            RiskLevel::Low => 0.0,
        };

        // 可验证性奖励：可验证的任务更值得用便宜模型
        let verifiability = profile.verifiability;

        let total = quality * self.quality_weight
            - cost_normalized * self.cost_sensitivity
            - latency_normalized * self.latency_sensitivity
            - risk * self.risk_penalty
            + verifiability * self.verifiability_bonus;

        ModelScore {
            model_key: model_key.to_string(),
            score: total,
            breakdown: ScoreBreakdown {
                expected_quality: quality,
                cost_term: cost_normalized * self.cost_sensitivity,
                latency_term: latency_normalized * self.latency_sensitivity,
                risk_term: risk * self.risk_penalty,
                verifiability_term: verifiability * self.verifiability_bonus,
            },
        }
    }
}
```

```rust
// orchestration/workflow_assembler.rs

/// 根据实时条件动态组装工作流链
///
/// 非静态 YAML 策略，而是每次请求根据：
/// 1. 任务特征（TaskProfile）
/// 2. 模型可用性（健康检查）
/// 3. 预算余量
/// 4. 历史表现
/// 动态决定调用哪些模型、用什么顺序、是否验证、是否升级
pub struct WorkflowAssembler {
    scorer: Arc<CostQualityScorer>,
    stats: Arc<StatsEngine>,
    health: Arc<ModelHealthChecker>,
}

pub struct DynamicWorkflow {
    pub workflow: Workflow,
    pub estimated_cost: f64,
    pub estimated_latency_ms: u64,
    pub reasoning: String,
}

pub enum WorkflowStep {
    /// 调用单个模型
    CallModel {
        model_key: String,
        temperature: f64,
    },
    /// 质量验证
    QualityCheck {
        threshold: f64,
        tools: Vec<String>,
    },
    /// 并行调用多个模型（DEBATE/MoA）
    ParallelCall {
        models: Vec<String>,
        merge_strategy: MergeStrategy,
    },
    /// Judge 仲裁
    Judge {
        judge_models: Vec<String>,
        aggregation: JudgeAggregation,
    },
    /// 随机抽检
    SpotCheck {
        inspector_model: String,
        probability: f64,
    },
    /// 人工闸门
    HumanGate {
        reason: String,
        timeout_secs: u32,
        fallback_action: FallbackAction,
    },
}

/// 类型化工作流枚举 — 每种策略是独立变体，可独立构建和测试
/// 替代 Vec<WorkflowStep> 索引跳转方案，消除索引偏移错误
pub enum Workflow {
    /// ROUTE: 单模型直出
    Route {
        model: String,
        temperature: f64,
    },
    /// CASCADE: 便宜 → 验证 → 升级，分支逻辑内置
    Cascade {
        first: CascadeStep,
        quality_check: QualityCheck,
        fallback: Option<Box<CascadeStep>>,
    },
    /// DEBATE: 多模型并行 → Judge → 可选抽检 → 可选人工闸门
    Debate {
        debaters: Vec<String>,
        judge: JudgeConfig,
        spot_check: Option<SpotCheckConfig>,
        human_gate: Option<HumanGateConfig>,
    },
    /// MoA: 多 Proposer → Aggregator
    MixtureOfAgents {
        proposers: Vec<String>,
        aggregator: String,
        quality_check: Option<QualityCheck>,
    },
}

pub struct CascadeStep {
    pub model_key: String,
    pub temperature: f64,
}

pub struct QualityCheck {
    pub threshold: f64,
    pub tools: Vec<String>,
}

pub struct JudgeConfig {
    pub judge_models: Vec<String>,
    pub aggregation: JudgeAggregation,
}

pub struct SpotCheckConfig {
    pub inspector_model: String,
    pub probability: f64,
}

pub struct HumanGateConfig {
    pub reason: String,
    pub timeout_secs: u32,
    pub fallback_action: FallbackAction,
}

pub enum MergeStrategy {
    BestOf,
    Aggregate,
    Vote,
}

pub enum FallbackAction {
    UseBestCandidate,    // 用 Judge 选出的最佳
    Abort,               // 放弃，返回错误
    EscalateToHuman,     // 转人工
}

impl WorkflowAssembler {
    /// 动态组装工作流（类型化枚举，无索引跳转）
    pub async fn assemble(
        &self,
        profile: &TaskProfile,
        budget_remaining: f64,
    ) -> Result<DynamicWorkflow> {
        // 选择初始模型（基于评分函数）
        let candidates = self.rank_models(profile, budget_remaining).await?;

        // 防护：无可用模型
        if candidates.is_empty() {
            return Err("NoHealthyModels: 所有模型不可用".into());
        }

        let workflow = match profile.complexity {
            // 低复杂度：单模型直出
            c if c < 0.4 => {
                let best = &candidates[0];
                Workflow::Route {
                    model: best.model_key.clone(),
                    temperature: 0.7,
                }
            }
            // 中复杂度：CASCADE（便宜 → 验证 → 升级）
            c if c < 0.7 => {
                let cheap = candidates.iter()
                    .find(|m| m.breakdown.cost_term < 0.1)
                    .unwrap_or(&candidates[0]);
                let mid = candidates.iter()
                    .find(|m| m.breakdown.expected_quality > 0.7)
                    .unwrap_or(&candidates.get(1).ok_or("No mid-tier model")?);

                Workflow::Cascade {
                    first: CascadeStep {
                        model_key: cheap.model_key.clone(),
                        temperature: 0.7,
                    },
                    quality_check: QualityCheck {
                        threshold: self.dynamic_threshold(profile).await,
                        tools: self.select_tools(profile),
                    },
                    fallback: Some(Box::new(CascadeStep {
                        model_key: mid.model_key.clone(),
                        temperature: 0.5,
                    })),
                }
            }
            // 高复杂度：DEBATE（多模型 + 多 Judge + 抽检）
            _ => {
                let debaters: Vec<String> = candidates.iter()
                    .take(3)
                    .map(|m| m.model_key.clone())
                    .collect();

                Workflow::Debate {
                    debaters,
                    judge: JudgeConfig {
                        judge_models: self.select_judges().await,
                        aggregation: JudgeAggregation::Consensus { threshold: 0.6 },
                    },
                    spot_check: Some(SpotCheckConfig {
                        inspector_model: "frontier".to_string(),
                        probability: 0.1,
                    }),
                    human_gate: if profile.risk == RiskLevel::Critical {
                        Some(HumanGateConfig {
                            reason: "极高风险任务，需人工确认".to_string(),
                            timeout_secs: 300,
                            fallback_action: FallbackAction::UseBestCandidate,
                        })
                    } else {
                        None
                    },
                }
            }
        };

        Ok(DynamicWorkflow {
            estimated_cost: self.estimate_workflow_cost(&workflow).await,
            estimated_latency_ms: self.estimate_workflow_latency(&workflow).await,
            reasoning: self.explain_workflow(&workflow, profile),
            workflow,
        })
    }

    /// 动态计算质量阈值
    ///
    /// 高风险任务：阈值提高（更严格）
    /// 高可验证性：阈值可以降低（因为验证工具能兜底）
    /// 预算充裕：阈值提高（可以多升级几次）
    async fn dynamic_threshold(&self, profile: &TaskProfile) -> f64 {
        let base = 0.65;

        let risk_adjustment = match profile.risk {
            RiskLevel::Critical => 0.15,
            RiskLevel::High => 0.10,
            RiskLevel::Medium => 0.05,
            RiskLevel::Low => 0.0,
        };

        let verifiability_adjustment = -profile.verifiability * 0.1;

        (base + risk_adjustment + verifiability_adjustment).clamp(0.3, 0.95)
    }
}
```

### 人工闸门（Human Gate）

```rust
// orchestration/human_gate.rs

/// 高风险 + 低共识时，暂停工作流等待人工确认
///
/// 触发条件（满足任一）：
/// 1. risk == Critical 且 Judge 共识 < 0.6
/// 2. 用户在 UI 中为该类任务开启了"人工审批"
/// 3. 成本超过 per_task_max_usd 的 80% 且仍未通过验证
pub struct HumanGate {
    db: Arc<SqlitePool>,
    timeout: Duration,
}

pub struct GateRequest {
    pub task_id: String,
    pub prompt_summary: String,     // 原始问题摘要
    pub candidates: Vec<CandidateAnswer>,
    pub judge_result: CrossJudgeResult,
    pub reason: String,
    pub created_at: DateTime<Utc>,
}

pub enum GateDecision {
    Approved { selected_idx: usize },   // 人工选择某个候选
    Revision { feedback: String },      // 要求重新生成
    Aborted,                             // 放弃本次请求
}

impl HumanGate {
    /// 发起人工闸门请求
    /// 1. 写入 gate_requests 表
    /// 2. 通过 Tauri IPC 通知 UI 显示审批面板
    /// 3. 等待用户操作或超时
    pub async fn request_review(
        &self,
        request: GateRequest,
    ) -> Result<GateDecision> {
        // 写入 DB
        self.persist_request(&request).await?;

        // 通知 UI（Tauri event）
        self.notify_ui(&request).await?;

        // 等待用户操作，带超时
        let decision = tokio::time::timeout(
            self.timeout,
            self.wait_for_decision(&request.task_id),
        ).await;

        match decision {
            Ok(d) => Ok(d),
            Err(_) => {
                // 超时：使用 fallback 策略
                Ok(GateDecision::Aborted)
            }
        }
    }
}
```

## 关键设计决策

### D1: 为什么答案质量比速度/成本更重要

开发者的痛点不是「AI 太贵」或「AI 太慢」，而是「AI 给的代码有时候是错的」。一个错误的代码提交可能花 2 小时 debug，远超省下的 API 费用。所以核心目标是**答案质量**。

### D2: 为什么 YAML + UI 双通道

不同用户偏好不同：
- 高级工程师：愿意改 YAML，需要 git diff 和版本控制
- 技术主管：想看可视化流程图，拖拽调参数
- 其他协作者：只看统计和成本，不碰配置

单通道只服务一种人。双通道服务所有人。

### D3: 为什么 Fork CC-Switch 而非全新项目

CC-Switch 已经有：
- 22+ shadcn/ui 组件（Button、Dialog、Card、Tabs、Switch 等）
- 完整的 Provider 管理 CRUD
- Axum 代理服务器（支持 SSE 流、故障转移、断路器）
- 使用统计仪表盘（recharts 图表）
- 国际化（i18next）
- 自动更新
- 暗色/亮色主题

从零做这些需要 6-8 周。Fork 后只需加编排引擎。

### D4: 为什么不是 Python 混合方案

用户不想安装 Python + Node + Rust。一个 Tauri 桌面应用 = 一个安装包。MiroFish 的模式是**移植为 Rust**，不是运行 Python 代码。

## 设计决策补充（审阅后修复）

### SSE 流式 UX 决策

采用**方案 C: ROUTE 流式、非 ROUTE 显示加载状态**。

- **ROUTE 策略**：直接流式透传，用户看到实时 token 输出（与直连模型体验一致）
- **CASCADE / DEBATE / MoA**：客户端显示加载指示器（"正在编排中..."），编排完成后一次性发送完整响应。原因：CASCADE 需要等第一级模型完成 + 验证通过后才能决定是否升级，无法做到真正的流式直通
- **CROSS-MODAL**：每个模态路径独立完成后合并，也是一次性发送

这对 Claude Code 的体验影响是：CASCADE 任务会有几秒到几十秒的等待，然后收到完整回答。作为补偿，UI 仪表盘实时显示编排进度（"第 1 级模型运行中..."、"质量验证中..."、"升级到第 2 级..."）。

### TaskClassifier 规格

**输入**：解析后的 Anthropic/OpenAI 请求体（messages 数组 + tools 数组 + model 字段）

**算法**：规则引擎（Phase 1），不使用 LLM 调用。基于以下启发式：

| 字段 | 推导规则 | 输出 |
|------|---------|------|
| `task_type` | 消息中包含代码块 (`\`\`\``) → `coding`；图片 content → `image`；短消息且无工具 → `summary` | enum |
| `complexity` | 消息总 token 数 / 4000（归一化到 0-1）；包含 tool_use → +0.3；多轮对话 → +0.2 | 0.0-1.0 |
| `risk` | 包含 "delete/remove/drop/truncate" → `critical`；包含 "refactor/migrate" → `high`；包含 "fix/update" → `medium`；其他 → `low` | enum |
| `verifiability` | 有 tool_use 且 tool 可自动运行 → 0.9；有 tool_use 但不可自动运行 → 0.5；无 tool → 0.1 | 0.0-1.0 |
| `has_image` | 消息中包含 image_url 或 image content block | bool |
| `need_code` | 消息中包含代码块或编程关键词（"implement/write/create function"） | bool |

**输出**：`TaskProfile` 结构体（YAML 策略匹配使用全部字段）

### 工具验证环境（收窄范围）

v2 MVP 的工具验证**不运行外部命令**（cargo test、eslint 等）。原因：网关没有用户项目目录的上下文，代码片段脱离项目无法运行有意义的测试。

**v2 验证方式**：
1. **结构化检查**：解析 LLM 输出的代码块，检查语法完整性（括号匹配、import 存在、函数签名闭合）
2. **LLM 自评**：用 judge 模型评估代码质量（0-1 分）
3. **模式匹配**：检查常见错误模式（空 catch、未处理异常、硬编码路径等）
4. **Schema 校验**：JSON 输出用 JSON Schema 验证

**v3+ 增强**：接受用户配置项目路径，在沙箱子进程中运行真实测试。这需要 IDE 集成或 CLI hook。

### 企业特性分阶段

**v2 (MVP)**：单用户桌面应用，不含团队管理。包含：预算追踪、成本仪表盘、策略配置。
**v5+ (Server 部署)**：多用户团队、RBAC、团队预算、API Key 共享。

### 热重载改进

使用 `notify` crate 的文件系统监听替代 5 秒轮询，加 2 秒去抖（等文件写入稳定后再重载）。

### YAML-UI 同步语义

**Single source of truth: YAML 文件。**

- YAML 文件是策略的唯一权威来源（磁盘即状态）
- UI 编辑器读写 YAML 文件：保存时写入磁盘，`notify` 触发热重载
- UI 打开时从 YAML 加载当前状态，不从内存缓存读取
- 冲突处理：如果 UI 编辑期间 YAML 被外部修改，UI 显示冲突提示并提供 diff/merge 选项

### UI 状态定义

编排面板的每个策略卡片需要处理以下状态：

| 状态 | 触发条件 | UI 表现 |
|------|---------|---------|
| **Idle** | 无活跃编排请求 | 显示策略参数和上次统计 |
| **Loading** | 编排请求进行中 | 进度指示器 + "正在编排中..." + 当前阶段文本 |
| **Active** | 编排请求正在执行策略 | 实时显示当前级别模型、验证结果 |
| **Success** | 编排完成，响应已返回 | 绿色勾 + 最终模型 + 耗时 + 成本 |
| **Error** | 编排过程中出错（模型不可用、预算超限等） | 红色错误信息 + 重试按钮 + 降级提示 |
| **Partial** | 部分验证通过但未达阈值，已升级到最终模型 | 黄色警告 + 升级次数 + 各级验证结果 |
| **Empty** | 无编排历史记录 | 引导用户开启编排 + 配置策略 |

### 边界条件

#### QualityGate 空工具列表

当 `verification.tools` 为空时，QualityGate 的 `avg` 计算会除以零（`inf >= 0.65` 恒为 true）。

```rust
// 修复：YAML 加载时验证
impl StrategyConfig {
    pub fn validate(&self) -> Result<()> {
        if self.verification.tools.is_empty() {
            anyhow::bail!("verification.tools 不能为空 — 至少需要一个验证工具");
        }
        Ok(())
    }
}
```

#### 预算竞态条件

多个并发请求同时通过预算检查，在任何一个完成扣费前都认为预算充足。

**修复：两阶段预算**

```rust
// 请求开始时：估算扣费（冻结额度）
let reserved = budget.reserve_estimated(task).await?;
// 请求完成时：按实际用量结算
budget.settle(reserved, actual_cost).await?;
```

`reserve_estimated` 基于历史数据估算本次请求成本（如按 token 数 × 模型单价），冻结该额度。多个并发请求共享同一预算池，冻结额度累加，超出限制则拒绝新请求。

#### 热重载中途配置切换

请求进行中的 CASCADE 升级可能因热重载导致策略配置变更（如升级目标模型被移除）。

**修复：请求持有配置快照**

```rust
// 每个请求开始时持有 Arc<StrategyConfig> 快照
let config_snapshot = self.config.read().await.clone();
// 请求全程使用快照，不受热重载影响
executor.execute(task, config_snapshot).await?;
// 热重载更新全局 config，但已有请求继续使用旧快照
```

10. 验证 tool-use 请求不被拦截
11. 验证热重载：修改 strategies.yaml 后策略自动更新
12. 验证开关：关闭编排后所有请求 passthrough

**Phase 1 完成标志：** Claude Code 通过网关发送代码请求 → 被正确路由到配置的模型 → 收到完整响应 → 延迟增加 < 100ms

### Phase 2 准备（Phase 1 完成后）

Phase 2 的核心是 CASCADE 策略 + QualityGate 增强。关键设计决策需要在 Phase 1 完成后确认：

1. **QualityGate 验证项权重** — 需要真实数据校准（结构化检查 vs LLM 自评 vs 模式匹配的权重分配）
2. **CASCADE 升级阈值** — 初始 0.65，Phase 3 后由 ThresholdOptimizer 自动调优
3. **温度衰减序列** — 初始 [0.0, 0.3, 0.7]，Phase 3 后根据 repair_loop 数据调整
