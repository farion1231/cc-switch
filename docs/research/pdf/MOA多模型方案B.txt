1. 理论依据：你的想法不是空想
A. FrugalGPT：低成本模型级联可以大幅降成本

FrugalGPT 提出三类降本策略：prompt adaptation、LLM approximation、LLM cascade。它的实验显示，模型级联可以在部分任务上接近最佳单模型表现，同时最高可报告到约 98% 成本降低，或者在相同成本下提升准确率。关键思想是：不是每个问题都需要最贵模型。

这正好对应你的设想：
简单任务 → 本地/便宜模型；
中等任务 → GLM / DeepSeek / MiniMax；
复杂任务 → GPT-5.5 / Claude Opus 4.7；
失败或不确定 → 再升级。

B. RouteLLM / Hybrid LLM：用路由器决定“该让哪个模型做”

RouteLLM 研究的是在强模型和弱模型之间学习路由，根据问题难度和偏好数据决定调用哪个模型，以优化成本和质量平衡。Hybrid LLM 也采用“根据问题难度分配给小模型或大模型”的方法。

你的系统里应该有一个独立模块，叫：

Model Router / Cost Router / check-model-skills

它不负责回答问题，只判断：

“这个任务是否已经拆到低成本模型可稳定执行？”

C. Cascade Routing：路由 + 级联可以合并成更优策略

2024 年的 Cascade Routing 研究把 routing 和 cascading 统一起来，指出成功的关键是质量估计器：系统必须能判断当前答案是否足够好，什么时候该升级模型。

这对你非常重要。你不能只设计“多个 agent 聊天”，而要设计：

任务 → 执行 → 质量估计 → 通过/重做/升级 → 最终交付

D. Mixture-of-Agents：多个模型多轮融合可以提升结果

Mixture-of-Agents 论文提出分层 MoA 架构，让多个模型的输出在下一层被其他模型吸收、改进；论文报告其 MoA 在 AlpacaEval 2.0、MT-Bench、FLASK 等基准上超过 GPT-4 Omni。

但要注意：2025 年也有研究提出 Self-MoA，有时“同一个强模型多次采样再融合”比混合多个不同模型更好，因为混入低质量模型会拉低平均质量。

所以结论是：

多模型融合有效，但不是模型越多越好。必须有筛选、排序、融合和早停。

E. LLM-Blender：先排序，再融合

LLM-Blender 的思路是先用 PairRanker 对多个候选答案排序，再用 GenFuser 融合 top-K 答案。

你的本地系统也应采用类似结构：

多个模型生成候选方案；
Judge / PairRanker 进行比较；
只保留 top 2–3；
再由强模型或融合模型生成最终答案。
F. Self-Consistency / Tree-of-Thought / ReAct：不是一次回答，而是搜索

Self-Consistency 通过采样多条推理路径再取一致答案，在多个推理基准上显著提升效果。Tree-of-Thought 进一步把推理变成树搜索，允许回溯、选择和前瞻。ReAct 把“推理”和“调用工具”结合起来，能减少幻觉和错误传播。

这说明：
你的系统不要让模型“一次性生成最终结果”，而要让它：

计划 → 执行 → 工具验证 → 反思 → 修改 → 再验证

5. 核心算法：成本-质量路由

你的系统要有一个统一评分函数：

Score(model, task)
= ExpectedQuality
- λ * Cost
- μ * Latency
- ρ * Risk
+ τ * ToolVerifiability

实际工程上可以这样做：

def route_task(task):
    features = extract_features(task)

    if task.risk in ["legal", "medical", "finance", "safety-critical"]:
        return "frontier_model_with_human_review"

    if task.requires_vision:
        return "vision_model"

    if task.is_simple_extraction and task.has_schema:
        return "cheap_local_model"

    if task.is_code and task.has_tests:
        return "cheap_code_model_then_test"

    if task.complexity_score > 0.75:
        return "frontier_planner"

    if task.uncertainty > 0.6:
        return "medium_model_or_ensemble"

    return "cheap_model"

一、核心结论

你的想法有理论依据，正确名称不是简单的“多 agent”，而是：

Multi-Model Agent System = 模型级联 + 动态路由 + 多候选融合 + 工具验证 + 记忆反馈 + 人工闸门

它的目标不是让低端模型“变成”Claude Opus 4.7，而是把工作变成流水线：

高端模型：拆解、规划、审查、关键判断
中端模型：复杂执行、长文写作、结构化生成
低端模型：批量执行、格式转换、局部代码、信息提取
本地工具：测试、编译、渲染、校验、引用、查错
Judge 模型：比较候选、发现问题、决定是否升级

真正能省钱的地方是：不要让顶级模型做所有 token 工作，只让它做最值钱的 5% 到 15% 的决策。

二、理论依据：为什么这种系统可行
1. FrugalGPT：模型级联可以显著降低成本

FrugalGPT 的核心是 LLM cascade：先用便宜模型回答，只有当结果不确定或不合格时才升级到更强模型。论文报告，在部分任务上可以接近最佳单模型表现，同时最多降低约 98% 推理成本；在相同成本下，也可能超过单一 GPT-4 的准确率。

这就是你的“低端模型先做，高端模型兜底”的理论基础。

2. RouteLLM：让路由器决定该用强模型还是弱模型

RouteLLM 研究的是用偏好数据训练路由器，动态选择强模型或弱模型，在尽量不损失质量的前提下降低成本；论文结果显示，在一些基准上可实现超过 2 倍成本节省。

你的系统必须有一个核心模块：

Model Router / check-model-skills：判断这个任务能不能交给低成本模型。

不是所有任务都交给便宜模型，而是先判断任务难度、风险、输出格式、可验证性。

3. Mixture-of-Agents：多个模型分层融合可以增强能力

Mixture-of-Agents 不是简单投票，而是多层结构：第一层多个模型生成答案，下一层模型读取上一层所有输出并改进。论文报告 MoA 在 AlpacaEval 2.0、MT-Bench、FLASK 等基准上取得很强结果，甚至在部分评测中超过 GPT-4 Omni。

但这也说明一个关键点：

多模型融合要有结构，不是群聊。

错误做法：

让 5 个 agent 自由聊天，最后自动总结

正确做法：

候选生成 → 排序 → 选择 top-K → 融合 → 验证 → 升级
4. LLM-Blender：先排序，再融合

LLM-Blender 提出 PairRanker 和 GenFuser：先比较多个模型输出，选出更好的候选，再融合优点生成最终答案。

这对你特别重要，因为低端模型输出很多时候是“部分对、部分错”。系统不能直接平均，而应该：

先判断哪个候选更可靠；
删除明显错误；
只融合 top 2 或 top 3；
最后再验证。
5. Tree-of-Thought：复杂问题不能一次生成，要搜索

Tree-of-Thought 把推理变成搜索过程：生成多个中间思路，自我评估，必要时回溯。论文显示，在需要规划和搜索的任务上，ToT 能显著提升表现，例如 Game of 24 任务中从普通 CoT 的低成功率提升到更高水平。

你的系统面对复杂问题时，不能让模型“一次回答完”，而应该：

生成多个方案 → 评估每个方案 → 选择路线 → 继续展开 → 回溯修正
6. ReAct：模型必须能调用工具，否则会幻觉

ReAct 的核心是 Reasoning + Acting：模型一边推理，一边调用外部工具，例如搜索、数据库、代码执行、环境反馈。论文指出 ReAct 在问答和事实验证任务中可以通过外部 API 降低幻觉和错误传播。

这正是你做本地工作台的关键：

模型不能只聊天，它必须能操作工具。

例如：

写代码后运行 pytest；
写 DOCX 后检查排版；
写 OpenSCAD 后渲染 PNG；
写 PCB 后跑 ERC / DRC；
写规格书后检查图片是否存在；
做市场研究后检查引用来源。
7. Reflexion：让 agent 从失败中改进

Reflexion 不是重新训练模型，而是把失败原因写入 episodic memory，下次尝试时参考。论文在 coding、语言推理、顺序决策任务上报告了显著提升。

这对你的工作台非常有用。比如你之前做规格书失败，系统应该记录：

失败经验：
- 不能让模型自由排版 PDF
- 必须先生成结构化 JSON
- 图片必须先做 asset manifest
- DOCX 必须基于模板渲染
- PDF 必须视觉检查缺图和错位

下一次执行时，系统自动避免同类错误。

8. SWE-agent：agent 成功不只靠模型，还靠“操作界面”

SWE-agent 研究发现，agent 与计算机交互的接口设计会显著影响软件工程任务表现；它通过专门的 agent-computer interface 帮助模型浏览仓库、编辑文件、运行测试。

这说明你的本地 agent 工作台不能只设计 prompt，还要设计：

文件系统接口；
工具执行接口；
日志接口；
测试接口；
回滚接口；
人工确认接口。

判断公式

你可以用这个评分：

CheapReadyScore =
0.25 * OutputClarity
+ 0.25 * Verifiability
+ 0.20 * ContextBoundedness
+ 0.15 * LowRisk
+ 0.15 * ToolSupport
- 0.30 * Ambiguity

规则：

Score >= 0.75：低端模型可执行
0.50 <= Score < 0.75：中端模型执行，强模型抽查
Score < 0.50：高端模型处理
高风险任务：无论分数多少，都要高端模型 + 人工确认
4. Model Router：模型路由器

路由器不能只看价格，要看：

特征	说明
complexity	任务复杂度
ambiguity	需求模糊度
risk	风险等级
verifiability	是否可自动验证
modality	文本、图像、语音、代码
context_length	上下文长度
required_format	是否需要 JSON / XML / DOCX
failure_history	此类任务过去失败率
budget	当前预算

伪代码：

def route(task):
    if task.risk in ["legal", "medical", "safety", "pcb_fabrication"]:
        return "frontier_model_plus_human_review"

    if task.requires_vision:
        return "vision_model"

    if task.requires_voice:
        return "voice_model"

    if task.output_format in ["json", "yaml", "csv"] and task.schema_available:
        if task.verifiability > 0.8 and task.ambiguity < 0.3:
            return "cheap_model"

    if task.type == "coding" and task.tests_available:
        return "cheap_or_medium_code_model_then_test"

    if task.complexity > 0.75 or task.ambiguity > 0.65:
        return "frontier_planner"

    return "medium_model"

九、本地框架怎么选
推荐总组合

LiteLLM 官方支持统一网关、重试、fallback、负载均衡、成本跟踪等能力，很适合作为多模型调用层。
LangGraph 官方强调 durable execution、checkpoint、human-in-the-loop、streaming，适合你这种长流程、本地工具、多阶段验证的 agent 系统。
Ollama 适合个人电脑快速运行本地模型，也支持接入 Claude Code、Codex、OpenCode 等工具链；vLLM 适合更生产化的本地推理服务，并提供 OpenAI-compatible server 和 structured outputs。

不建议一开始就重度绑定 AutoGen

AutoGen 本身是经典多 agent 框架，但它的 GitHub 页面显示 AutoGen 进入 maintenance mode；微软新的方向是 Microsoft Agent Framework，它结合了 AutoGen 与 Semantic Kernel 的能力。

CrewAI 可以用，但更适合“角色协作”和快速原型；它的 Flows 更适合精确流程控制，Crews 更适合动态协作。

我的建议：

主框架：LangGraph
模型网关：LiteLLM
本地模型：Ollama / vLLM
可选角色协作：CrewAI
不建议主系统依赖旧 AutoGen

十七、Agent 设计模板
1. Chief Planner Agent

职责：

- 只做任务拆解
- 不直接写大段最终内容
- 输出任务图
- 标记风险
- 指定验证方式

系统提示词：

你是 Chief Planner。你的任务不是直接完成用户任务，而是把任务拆解成可验证、可委托、可回滚的子任务。

必须输出：
1. 任务目标
2. 子任务列表
3. 每个子任务的输入、输出、成功标准
4. 推荐模型等级：local / cheap / medium / frontier
5. 验证工具
6. 风险等级
7. 是否需要人工确认

禁止：
- 不允许把模糊任务直接交给低成本模型
- 不允许省略验证步骤
- 不允许让模型自由生成最终工程交付物
2. check-model-skills Agent
你是 check-model-skills。你的唯一职责是判断某个子任务是否可以交给低成本模型执行。

判断维度：
- 任务是否清晰
- 输入是否完整
- 输出是否有固定格式
- 是否有 schema / 测试 / 工具验证
- 是否有安全或商业风险
- 上下文长度是否适合低端模型
- 失败后是否可重试
- 是否需要视觉、语音、代码、文件操作

输出 JSON：
{
  "can_use_cheap_model": true/false,
  "recommended_model_tier": "...",
  "risk_level": "...",
  "missing_information": [],
  "required_constraints": [],
  "verification_steps": [],
  "fallback_plan": {}
}
3. Cheap Executor Agent
你是执行模型。你只完成指定子任务，不扩展范围，不做未经要求的推断。

规则：
- 只输出指定格式
- 不编造缺失信息
- 缺失字段写 TBD
- 不改变字段名
- 不解释过程
- 不添加多余内容
4. Verifier Agent
你是质量审查模型。你的任务是找错，不是润色。

检查：
- 是否满足用户要求
- 是否违反 schema
- 是否有缺失字段
- 是否有编造
- 是否有单位错误
- 是否有格式错误
- 是否需要升级模型
- 是否需要人工确认

输出：
{
  "pass": true/false,
  "issues": [],
  "severity": "low/medium/high",
  "repair_instruction": "...",
  "should_escalate": true/false
}

第一选择：直接 Fork ec-switch

这是最接近你目标的项目。它本身就是一个跨平台桌面 App，用来管理 Claude Code、Codex、Gemini CLI、OpenCode、OpenClaw、Hermes Agent 等工具。项目是 MIT license，技术栈是 Tauri 2.8 + Rust 后端、React + TypeScript 前端，并且已有本地 proxy、provider 管理、MCP 管理、使用统计、会话管理等模块。

下载：

git clone https://github.com/farion1231/cc-switch.git
cd cc-switch
pnpm install
pnpm dev

它的本地路由机制已经非常接近你的目标：开启 routing 后，它会把 Claude Code 的 ANTHROPIC_BASE_URL 改成本地地址，把 Codex 的 base_url 改成 http://127.0.0.1:15721/v1，然后本地 routing service 负责识别请求来源、选择 provider、转发请求、记录日志，并支持 failover。

所以如果你要做“类似 EC Switch 的本地网关”，最优路线不是重新造 UI，而是：

Fork ec-switch，然后重点增强它的 Router / ProxyService / Usage Statistics / Evaluation Harness。

你要加的不是普通模型切换，而是：

智能路由：
- background → 本地 Qwen / Ollama
- simple edit → DeepSeek / GLM / Qwen
- long context → Gemini / Claude / GPT
- thinking → GPT-5.5 / Claude Opus
- failed twice → 自动升级
- high-risk change → 强模型审查
第二选择：claude-code-router 作为 Claude Code 路由核心

如果你暂时不想做桌面 App，只想先测试 Claude Code 多模型路由，musistudio/claude-code-router 是最合适的底座。它明确支持 Claude Code 请求路由、多 provider、请求/响应转换、/model 动态切换、CLI 管理和插件系统。

下载：

git clone https://github.com/musistudio/claude-code-router.git
cd claude-code-router
npm install
npm run build

也可以直接全局安装：

npm install -g @musistudio/claude-code-router

它已经支持 OpenRouter、DeepSeek、Ollama、Gemini、Volcengine、SiliconFlow 等 provider，并且配置里可以把 background、think、longContext 路由到不同模型。项目示例里就有 DeepSeek、Ollama、Gemini、Qwen、GLM、Claude 等配置方式。

它适合做：

Claude Code → 本地 Claude-compatible Router → 不同模型

但它不是完整的“Claude Code + Codex + Gemini CLI 全统一桌面网关”。所以它更适合作为你的 Claude Code 路由模块参考。

第三选择：LiteLLM 作为底层统一模型网关

LiteLLM 是最适合作为你网关底层的模型访问层。它是开源 AI Gateway，提供统一接口调用 100+ LLM provider，包括 OpenAI、Anthropic、Gemini、Bedrock、Azure 等，并使用 OpenAI format。

下载：

git clone https://github.com/BerriAI/litellm.git
cd litellm

快速启动 proxy：

pip install -U "litellm[proxy]"
litellm --config litellm_config.yaml --port 4000

它适合负责：

- provider 统一
- API key 管理
- fallback
- budget
- logging
- rate limit
- cost tracking
- OpenAI-compatible endpoint

但是 LiteLLM 本身不是专门为 Claude Code / Codex 做桌面配置切换的。所以最佳组合是：

第四选择：claude-code-proxy 作为 Anthropic ↔ OpenAI 协议转换参考

1rgs/claude-code-proxy 是一个更小的项目，目标很明确：让 Anthropic clients，例如 Claude Code，可以用 Gemini、OpenAI 或 Anthropic 后端，并且通过 LiteLLM 做转换。

下载：

第五选择：官方 openai/codex

Codex 本身已经开源，Apache-2.0 license，是本地运行的 coding agent。官方 README 明确说 Codex CLI runs locally on your computer，并支持 npm、Homebrew、脚本安装。

下载：

git clone https://github.com/openai/codex.git
cd codex

安装使用：

npm install -g @openai/codex
codex

Codex 官方配置已经支持 custom model providers。你可以在 ~/.codex/config.toml 定义自定义 provider，设置 base_url、env_key、wire_api，再把 Codex 指向你的本地 gateway。

示例：

model = "gpt-5.5"
model_provider = "local_gateway"

[model_providers.local_gateway]
name = "My Local AI Gateway"
base_url = "http://127.0.0.1:15721/v1"
env_key = "LOCAL_GATEWAY_API_KEY"
wire_api = "responses"

或者如果你的本地网关只实现 Chat Completions：

model = "qwen-coder-local"
model_provider = "local_gateway_chat"

[model_providers.local_gateway_chat]
name = "My Local Chat Gateway"
base_url = "http://127.0.0.1:15721/v1"
env_key = "LOCAL_GATEWAY_API_KEY"
wire_api = "chat"

注意：Codex 的 custom provider 不能复用内置 provider ID，例如 openai、ollama、lmstudio；你要用自己的名字，例如 local_gateway。

第二阶段：加智能路由

你要在 ec-switch 的 ProxyService 或单独新建 SmartRouterService。

路由策略建议：

routes:
  default:
    provider: litellm
    model: deepseek-chat

  background:
    provider: ollama
    model: qwen2.5-coder:latest

  simple_edit:
    provider: litellm
    model: deepseek-chat

  test_generation:
    provider: litellm
    model: qwen-coder

  thinking:
    provider: litellm
    model: claude-opus-4.7

  long_context:
    provider: litellm
    model: gemini-pro-long-context

  final_review:
    provider: litellm
    model: gpt-5.5

rules:
  - if: input_tokens > 60000
    route: long_context

  - if: request_contains in ["think", "plan", "architecture", "review"]
    route: thinking

  - if: tool_call_count > 20
    route: final_review

  - if: previous_failures >= 2
    route: final_review

  - if: app == "codex" and task_type == "small_edit"
    route: simple_edit

  - if: app == "claude-code" and task_type == "background"
    route: background
第三阶段：加评估系统
